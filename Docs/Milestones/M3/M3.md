# M3 Generator Pipeline

本文件是 M3 阶段的“作战图”。它把 `Docs/Milestones.md` 中 M3 的目标拆成可勾选的最小项，并把 M3 引入的关键设计决策（特别是“结构化内容图 / Theme 输入数据 / 站点级功能性产物 / 本地静态输出 / 实时模式 / 源工程 vs 构建产物边界”）固化下来，作为后续阶段不可回退的护栏。

> 上下文恢复入口：阅读顺序为 `Docs/Architecture.md` → `Docs/Milestones.md` → `Docs/Milestones/M1/M1.md` → `Docs/Milestones/M2/M2.md` → 本文件。
>
> M3 的任何实现都必须与这里的“关键决策”一致；如需调整，先改本文件再改代码。

## 1. 目标

让 Bocchi 真正“构建”自己的站点：把 M2 扫描出来的强类型内容文档汇总为一份**标准化内容图（Content Graph）**，进而：

1. 写出 **Theme 输入数据**（`<workspace>/.bocchi/input/*.json`）：稳定、向后兼容、富元数据；同时附带正文的 `markdown` / `html` / `excerpt` 三态表达，把 Theme 阶段的二次处理空间留给 Theme（代码高亮、图片放大、表格、数学公式、视音频、配置化页面 …）。
2. 写出 **站点级功能性产物**：`robots.txt`、`sitemap.xml`、RSS Feed（`feed.xml`）。这些与具体 Theme 无关，完全由后台配置 + 内容图决定。
3. 写出 **本地静态产物**（`<workspace>/output/public/`）：媒体复制、功能性页面、构建 manifest。这个目录可以被 Push 给 GitHub Pages / Nginx / Vercel / Cloudflare Pages 等任意静态托管。
4. 同时支持 **实时模式**：Home Server 通过 HTTP 端点直接吐出与上述 JSON / Media / 功能性页面**完全一致**的内容，用于本地实时预览、编辑器预览、跨进程调用等场景。
5. 在 Home Server Dashboard 中提供 **触发 Full Build / 查看构建日志 / 查看 manifest / 下载静态包 (zip)** 的最小可用入口。
6. 为后续 Theme 阶段铺路：M3 的所有输出在“数据形状”上必须足够供 M5 默认 SvelteKit Theme 直接消费，无需 Theme 自行再做一次扫描或一次解析。

## 2. 范围与非目标

### 2.1 在范围内

- **Generator Pipeline 框架**：可组合的阶段（Stage）、构建日志、阶段化错误聚合、可取消、可被 CLI / 后台 / 测试驱动。
- **Content Graph 构建**：把 `ScanResult` 加工为标准化、跨内容引用、按时间 / 标签 / 分类 / 年份索引的视图。
- **媒体重写**：把"相对源文件"的媒体引用改写为"站点根相对"的稳定路径；产出 `MediaAsset` 列表（包含源绝对路径 / 站点路径 / hash / 内容类型）。
- **Theme 输入数据写入**：`site.json` / `navigation.json` / `posts.json` / `pages.json` / `works.json` / `notes.json` / `friends.json` / `photos.json` / `theme-config.json` / `build-context.json`；JSON Schema 与版本号显式记录在数据头部。
- **站点级功能性产物**：`robots.txt`、`sitemap.xml`、`feed.xml`（Atom 1.0）。
- **媒体复制**：原始媒体按规范化路径直接复制（不做任何派生 / 不依赖 Theme）。M3 不引入 webp / 缩略图（衍生媒体目录 `.bocchi/cache/derivatives/` 预留位置，真正落地随媒体处理里程碑）。
- **构建 manifest**：`output/public/.bocchi-manifest.json` 列出每个输出文件的路径、大小、SHA-256、生成阶段，便于发布对账与未来增量。
- **输出目录校验**：完整 Full Build 末尾对 `output/public/` 做一遍校验（路径穿越防护、孤儿文件清理、manifest 一致性）。
- **指纹与增量基础**：每次构建以 `BuildFingerprint`（来自所有源文件 hash 的有序汇总）入库；同指纹直接跳过整次构建（最低层级“is-up-to-date”），细粒度增量留给后续。
- **实时模式 HTTP 端点**：`/_bocchi/preview/data/{name}.json`、`/_bocchi/preview/media/{*path}`、`/_bocchi/preview/robots.txt`、`/_bocchi/preview/sitemap.xml`、`/_bocchi/preview/feed.xml`。实时模式与 Full Build 共享同一套 Pipeline，二者**唯一区别**是 Sink。
- **Home Server 入口**：`/build` 页面（触发 Full Build、显示阶段日志、列出 manifest、下载 zip 包）；首页增加构建入口。
- **CLI 入口**：`dotnet run --project Src/HomeServer/Bocchi.HomeServer -- build` 命令行触发 Full Build（不启动 Web）。
- **测试**：Pipeline / ContentGraphBuilder / 媒体改写 / Theme 输入写入 / 功能性产物 / 文件系统 Sink / 实时端点 / Home Server `/build` 页面与 zip 下载。
- 在 Architecture / Milestones 总文档中固化决策。

### 2.2 非目标（移交给后续里程碑）

- M4：内容列表 / 编辑表单 / Theme 配置 UI（schema → 表单）/ 构建日志页的"实时流式"体验（M3 只交付最小同步页）。
- M5：默认 SvelteKit Theme 的具体实现；Theme 真正消费 `.bocchi/input/` 并把 HTML 写入 `output/public/`。在 M5 落地之前，M3 的 Full Build 在"Theme 渲染"这一步**不报错地跳过**：因为当下无可用 Theme，Pipeline 仅写到 Theme 输入即止；这是 Pipeline 阶段化的自然语义，不是过渡方案。
- M6：发布目标（Local Directory / Cloudflare Pages 自动化 / 发布历史）、静态搜索索引（自研 JSON index）。M3 仅产出"可发布的静态目录"作为本地基准。
- 媒体衍生（webp、缩略图、视频转码）：交给专门的媒体处理里程碑，目录 `.bocchi/cache/derivatives/` 已留位。
- 细粒度增量构建（artifact 级依赖图，"改一篇 → 只重写一篇 + RSS + 两个目录页"）：M3 仅做 **build-level 指纹判定**（同指纹整次跳过）；artifact 级依赖图的设计方案见 §7，落地推迟到 M6 与发布管线一起设计，避免在没有真实发布场景前过早设计。
- Git 远程接入（push/pull / GitHub）：依旧推迟到 M6 与发布管线一起设计。
- 评论 / 动态短文 / Webmention / 统计：M7 Cloud Server 范围。

## 3. 关键决策

这些决策在 M3 落定，后续不再轻易回退；如需调整，必须先改本节并同步 `Docs/Architecture.md`、`Docs/Milestones.md`。

### 3.1 输出目录与"源工程 vs 构建产物"边界（最核心）

- 内容空间 `<workspace>/content/` 是用户的源工程；**Generator 与 Theme 的所有输出都不允许出现在内容空间内**。M2 已确立此切分（§3.1），M3 继续严格遵守。
- M3 的输出全部落入 **Bocchi 系统空间**：
  - `.bocchi/input/`：Theme 输入数据（JSON）。由 M3 写入，由 M5 Theme 消费。
  - `.bocchi/cache/derivatives/`：衍生媒体（M3 留位，不写入）。
  - `output/public/`：可发布的静态目录（media / robots / sitemap / feed / manifest / Theme HTML）。
- 例外：**修正性回写**到内容空间是允许的，但限于"扫描器发现元数据不完善 / 不规范"的修复（M3 暂不主动触发；保留接口约定）。任何修正性回写必须：
  1. 由用户在 Dashboard 主动确认（M4 落地 UI；M3 仅约定 API）。
  2. 产生一次 Git diff，便于用户审阅与回滚。
  3. 不修改 Markdown 正文，只修订 frontmatter 字段。
- 路径全部经由 `WorkspaceLayout` / `ContentSpaceLayout` 提供，禁止 Generator 内任何地方手拼路径。

### 3.2 构建模型：Pipeline + Stage + Sink

| 抽象 | 职责 |
| --- | --- |
| `IBuildStage` | 一个可组合的最小阶段（例如 `LoadContentStage` / `BuildContentGraphStage` / `WriteThemeInputStage` / `WriteSiteArtifactsStage` / `CopyMediaStage` / `WriteManifestStage` / `RunThemeBuildStage`）。每个阶段接收 `BuildSession`，向其追加日志 / artifact，必要时取消。 |
| `BuildSession` | 单次构建的可变上下文：`SessionId`、`Mode`（FullBuild / Live / DryRun）、`StartedAt`、`Configuration`、`ContentSpace` 快照、`Logs`、`Artifacts`、`Fingerprint`、`Cancellation`。 |
| `IBuildSink` | 单次构建的"写出端"。M3 实现两种：`FileSystemBuildSink`（写 `output/public/` 与 `.bocchi/input/`）、`HttpStreamBuildSink`（实时模式下，把 artifact 序列化到 HTTP 响应）。`DryRunBuildSink` 仅断言不写。 |
| `GeneratorPipeline` | 编排器：固定阶段次序，串行执行；任何阶段抛出 `BuildException` 时短路并归档日志；最终产出 `BuildResult`。 |
| `BuildArtifact` | 单个输出项：`Path`（站点根相对 + `'/'` 归一化）/ `ContentType` / `SizeBytes` / `Sha256` / `ProducedBy`（阶段名）/ `Source`（产物字节流或本地源文件）。 |

关键护栏：**实时模式与本地静态模式共享同一套 Pipeline 与 Stage**，二者唯一差异在 `IBuildSink`。这是用户提出的"千万不能用本地起服务 + 爬虫保存 HTML"问题的架构级答案——M3 不存在"先起一个 Web 再爬"的代码路径，因为两条路都从同一组结构化 artifact 出发。

### 3.3 Content Graph：站点视角的标准化视图

`ContentGraph` 是只读、不可变的快照，由 `ContentGraphBuilder` 从 `ScanResult` 一次性构建。它承担"把分散的内容文档汇总为站点视角"的全部职责，是 M3 之后所有下游写出器的唯一事实来源。

`ContentGraph` 包含：

- `Site`：站点设置 + 派生信息（解析后的 `BaseUrl`、`Author`、`Navigation`、功能开关）。
- `Posts`：按 `publishedAt desc` 稳定排序，每项含 slug / title / publishedAt / updatedAt / category / tags / summary / cover / 正文 `markdown` + `html` + `excerpt` / 媒体引用列表（已站点化路径）/ 站点 URL（如 `/posts/2026/hello-bocchi/`）。
- `Pages`：按 `order asc, title asc`。
- `Works`：按 `featured desc, publishedAt desc`（M3 中 Work 暂无 publishedAt，故落到 `year desc, title asc`）。
- `Notes`：按 `publishedAt desc`。
- `Friends`：按 `order asc, name asc`。
- `Photos`：M3 占位空列表。
- `MediaAssets`：去重后的全站媒体清单，每项 `SourceAbsolutePath` / `SiteRelativePath` / `Sha256` / `ContentType` / `OwnerContentId?`。
- `Indices`：派生索引（按年份分组、按标签分组、按分类分组、最新 N 项视图）。这些索引由 Builder 一次性算出，写出器直接用，避免 Theme 各自重复实现。
- `Diagnostics`：图构建期间的告警 / 信息（与 `ScanResult.Errors` 区分；扫描产生的错误如果阻塞图构建会被升级）。

不变量：

- 同一 slug 同一 kind 唯一；冲突时构图失败并返回 `BuildException`。
- 媒体引用必须有对应 `MediaAsset`；找不到时是 `Error`，构图阻塞。
- 草稿（`status: draft`）默认 **不进入** `ContentGraph`；可通过 `BuildOptions.IncludeDrafts` 在实时模式中显式开启。

### 3.4 站点 URL 与媒体路径方案

| 资源 | 站点路径（`baseUrl` 之下） |
| --- | --- |
| 文章 | `/posts/{year}/{slug}/` |
| 独立页面 | `/{slug}/`（首页 `/`、`/about/` 等） |
| 作品 | `/works/{year}/{slug}/` |
| 短文 | `/notes/{year}/`（聚合页；M3 仅约定 URL，不写 HTML，HTML 由 Theme 决定是否单条详情） |
| 友链 | `/friends/` |
| 媒体（属于具体一篇） | `/media/{kind}/{year}/{slug}/{filename}` |
| 媒体（独立页面） | `/media/pages/{slug}/{filename}` |
| 媒体（友链头像等单文件源） | `/media/friends/{filename}` |
| RSS | `/feed.xml` |
| Sitemap | `/sitemap.xml` |
| Robots | `/robots.txt` |
| Theme 输入 JSON | 不在站点输出树中；位于 `.bocchi/input/{name}.json` |
| Build manifest | `/.bocchi-manifest.json`（在 `output/public/` 根；零信息泄露，不含本机路径） |

理由：媒体路径以"内容所属 + slug"为前缀，保证 slug 唯一时 URL 也唯一；不与 Theme 渲染细节耦合。所有路径以 `'/'` 开头、`'/'` 归一化、UTF-8 路径仅允许 `[a-zA-Z0-9._~/-]` 加 percent-encoded（M3 不做转译，由 slug 校验在 M2 ContentScanner 保证）。

### 3.5 Theme 输入 JSON 契约

每个 `.bocchi/input/*.json` 顶层包含：

```json
{
  "$schema": "https://bocchi.local/schema/v1/<name>.json",
  "contractVersion": "1.0",
  "generatedAt": "2026-05-14T08:57:35Z",
  "data": { ... }
}
```

- `contractVersion` 与 `ThemeContractVersion.V1` 一致；将来不兼容变更升 major，向后兼容增字段。
- `data` 形状对应该 JSON 的"主体"。Post / Page / Work / Note 数组中的每条都同时携带 `markdown`、`html`、`excerpt` 三个正文表达，由 Theme 自行选择（应对代码高亮、数学公式、图片放大、视音频等 Theme 专属处理需求）。
- 媒体路径写入时**已被站点化**（§3.4）；Theme 不需要重新解析"相对源文件"路径。
- `build-context.json` 即 `Bocchi.GeneratorContract.BuildContext` 的 JSON 表达（已存在），追加 `bocchiVersion` 字段。
- JSON 序列化统一走 `System.Text.Json`，`PropertyNamingPolicy = CamelCase`，`WriteIndented = true`（人类可读，Git diff 友好）；不写出 BOM。

### 3.6 站点级功能性产物

| 产物 | 决策 |
| --- | --- |
| `robots.txt` | 完全由 `SiteSettings` + Dashboard 配置决定。M3 默认：`User-agent: *` + `Allow: /` + `Sitemap: {baseUrl}/sitemap.xml`。可通过 `SiteSettings.EnableSitemap=false` 关闭 sitemap 行；可通过 `SiteSettings.Robots`（M3 新增字段：`Allow` / `Disallow` 列表）覆盖。 |
| `sitemap.xml` | XML Sitemap 0.9。包含 Posts、Pages、Works 的稳定 URL（草稿排除）。`lastmod` 取 `updatedAt ?? publishedAt`，缺失时回退到源文件 mtime。`changefreq` 与 `priority` 不主动写出（避免假信号）。`SiteSettings.EnableSitemap=false` 时阶段直接跳过且 manifest 不含此文件。 |
| `feed.xml` | Atom 1.0（不是 RSS 2.0）。Atom 时间字段语义更严格、id 必填、与 SemVer 思维更一致。包含最新 N 条 Posts（N 由 `SiteSettings.FeedItemCount` 决定，默认 20）。每条 `<entry>` 含 `id`（URN tag scheme：`tag:{host},{publishedDate}:posts/{year}/{slug}`）、`title`、`updated`、`published`、`summary`、`content type="html"`、`category`。`SiteSettings.EnableRss=false` 时跳过。 |
| 媒体复制 | 二进制流式复制；保留 mtime；目标路径来自 `MediaAsset.SiteRelativePath`。 |

### 3.7 Pipeline 阶段顺序（固定）

固定次序、显式装配、可单元测试。任何"想插一个中间步骤"的需求都必须新增一个 Stage，不允许就地修改已有 Stage 的内部职责。

```
 1. PrepareSessionStage             (建立 BuildSession + 日志器)
 2. LoadContentStage                (调用 ContentScanner.ScanAsync，把 ScanResult 注入 session)
 3. BuildContentGraphStage          (ContentGraph + Diagnostics)
 4. ComputeFingerprintStage         (BuildFingerprint = SHA-256(sorted(SourceHash)) + 配置 hash)
 5. ShortCircuitIfUpToDateStage     (Mode=FullBuild 且与上次成功构建指纹一致时，跳过余下阶段)
 6. WriteThemeInputStage            (写出 .bocchi/input/*.json)
 7. WriteSiteArtifactsStage         (robots.txt / sitemap.xml / feed.xml)
 8. CopyMediaStage                  (站点化媒体)
 9. RunThemeBuildStage              (如果 ThemeRunner 可用，执行 Theme 构建；否则记录 Skipped)
10. WriteManifestStage              (output/public/.bocchi-manifest.json)
11. ValidateOutputStage             (路径穿越 / 孤儿文件 / manifest 一致性)
12. CommitSessionStage              (落库：BuildRuns / BuildArtifacts)
```

实时模式（`BuildMode.Live`）只跑 1-4、6-9（不 short-circuit、不写 manifest、不 validate、不持久化），并由 `HttpStreamBuildSink` 把 artifact 流式吐给 HTTP 响应。

### 3.8 Theme 运行（M3 行为）

- 引入 `IThemeRunner` 抽象：根据 `ThemeManifest`（来自 `<workspace>/themes/<themeId>/theme.json`）执行 `installCommand`（首次）与 `command`。
- M3 默认实现 `ThemeRunner`：纯 `System.Diagnostics.Process` 包装；环境变量注入 `BOCCHI_INPUT_DIR`、`BOCCHI_OUTPUT_DIR`、`BOCCHI_THEME_ID`、`BOCCHI_BASE_URL`、`BOCCHI_ENVIRONMENT`；工作目录为 Theme 根；超时 / 取消 / stdout+stderr 日志原样转发到 `BuildSession`。
- M3 仅实现到这一层。在 `<workspace>/themes/` 为空 / 默认主题 id 未配置时，`RunThemeBuildStage.Status = Skipped`，构建仍然成功（产出 Theme 输入 + 功能性产物 + 媒体）。
- M5 落地默认 SvelteKit Theme 时，沿用 `IThemeRunner` 与此环境约定；不需要再引入新概念。

### 3.9 持久化（SQLite 扩展）

M3 在 `bocchi.sqlite` 中新增 user_version 2 → 3 迁移：

- `BuildRuns`：`Id`（PK）/ `Mode` / `Status`（Running / Succeeded / Failed / Skipped）/ `StartedAt` / `FinishedAt?` / `Fingerprint` / `Reason?`（失败原因首句 / Skipped 原因）/ `ScanRunId`（外键，可空）。
- `BuildArtifacts`：`Id` / `BuildRunId`（外键）/ `Path` / `ContentType` / `SizeBytes` / `Sha256` / `ProducedBy`。
- `BuildStageLogs`：`Id` / `BuildRunId` / `Stage` / `Level`（Info / Warning / Error）/ `Message` / `OccurredAt`。

迁移使用 M2 已有的 `SchemaMigrator.RegisterMigration` 风格（递增版本号 + 幂等 DDL）。

### 3.10 搜索：自研 JSON Index vs Pagefind（决策）

- **决策：默认搜索方案采用"由 Generator 输出的自研 JSON Index"。本节在 M3 中关闭 `Docs/Milestones.md` 待决问题。Pagefind 不进入默认路线。**
- 理由：
  1. 自研 JSON Index 完全位于已有架构内："Generator 输出结构化数据 → Theme 消费"。Pagefind 在 Theme 渲染**之后**对 HTML 进行二次扫描，违反"先有结构化数据再有 HTML"的方向。
  2. Pagefind 引入一个 Rust 二进制 + Node CLI 依赖，把搜索索引的可用性绑定到 Theme 构建链；对"Theme 可以是非 Node 的任何技术栈"这一架构开放性产生反向耦合。
  3. M3 当前内容量级（个人主页）下，自研 JSON Index 的体积与体验完全够用；体验天花板更高时仍然可以**额外**支持 Pagefind 作为可选 sink，不需要替换默认路线。
- 落地：**M3 不实现搜索索引**（M6 范围）。M3 仅保证 Theme 输入数据中含有足够多的字段（title / slug / summary / excerpt / tags / category / publishedAt），让 M6 自研 JSON Index 可以从 `.bocchi/input/posts.json` 等直接构建，无需重扫内容空间。

### 3.11 Cloudflare Pages（决策）

- **决策：M3 不接入 Cloudflare Pages 自动化；以"本地静态目录 + zip 包"为基准产物（用户在外部用任意手段 Push）。本节在 M3 中关闭 `Docs/Milestones.md` 待决问题。**
- 理由：M3 的目标是"得到一个可发布的本地静态目录"。这一目录可以用任何手段 Push 到 Cloudflare Pages / GitHub Pages / Nginx / Vercel。把发布管线（凭据、Wrangler、webhook、发布历史）拉到 M3 会让 M3 失焦。M6 在已有可发布目录的基础上接入自动化是自然的演进。

### 3.12 增量构建（决策）

- **决策：M3 实现"build-level 指纹判定"作为最低增量层级；artifact 级依赖图设计已记录但不在 M3 落地。**
- 用户提出的"改一篇文章只影响 文章页 / 目录页 / 首页 / RSS"诉求是合理的；但完整落地需要先有：
  - artifact ⇄ 源集合的反向映射（设计已在 §3.7 / §3.9 表结构中预留扩展位）；
  - Theme 侧合作的 partial-build 协议（Theme 必须支持"仅重渲染受影响页面"）。
- 这两者都涉及 M5 Theme 的具体形态。M3 现在落地完整 artifact 级增量等于在 Theme 真实形态出现之前先设计协议，违反"不做过渡方案"的原则。
- 因此 M3 的增量边界明确为：**同指纹 → 整次 skip；指纹变化 → 整次 rebuild**。这一行为对单用户日常写作够用（不到秒级），同时把"细粒度增量"这件事完整推到 Theme 真正存在之后。

### 3.13 工程层决策

| 主题 | 决策 |
| --- | --- |
| 项目布置 | 新增 `Src/Core/Bocchi.Generator/`（实现）；契约类型仍住在 `Bocchi.GeneratorContract`（扩展，不破坏 v1）。`Bocchi.Generator` 依赖 `Bocchi.Workspace` + `Bocchi.ContentModel` + `Bocchi.GeneratorContract`。 |
| HomeServer 依赖 | `Bocchi.HomeServer` 新增引用 `Bocchi.Generator`；DI 通过 `AddBocchiGenerator(IConfiguration)` 扩展接入。 |
| 配置节 | `appsettings.json` 新增 `Bocchi:Generator`（`DefaultThemeId?` / `Environment`（默认 `production`）/ `IncludeDrafts`（默认 `false`） / `FeedItemCount` 默认 20 / `FailOnContentError`（默认 `true`））。 |
| JSON 序列化 | `System.Text.Json`，CamelCase，缩进，UTF-8 无 BOM。共用 `JsonSerializerOptions` 单例。 |
| 异常类型 | 每个失败模式有专属异常：`ContentGraphException`（构图阶段）、`BuildPipelineException`（流水线 / Sink）、`ThemeRunnerException`（外部 Theme 进程）。统一基类 `GeneratorException`。 |
| 时间源 | 一律 `TimeProvider`，禁止 `DateTime.UtcNow` / `DateTimeOffset.Now` 直接使用。 |
| 异步 | IO（写文件 / SQLite / Process）全部异步；CPU 密集（HTML 渲染、XML 序列化）保持同步即可。`CancellationToken` 全链路必须接住。 |
| 路径处理 | 写文件前一律用 `Path.GetFullPath` 计算绝对路径，且断言绝对路径以输出根为前缀；任何不通过则抛 `BuildPipelineException`（防穿越）。 |
| 静态/实时端点 URL 前缀 | 实时端点统一在 `/_bocchi/preview/` 之下，独立于 Theme 任何可能的 URL；不会与未来 Theme HTML 输出冲突。 |
| 字符串/编码 | 所有文本产物（JSON / XML / robots）UTF-8 无 BOM；XML 显式声明 `version="1.0" encoding="utf-8"`。 |
| 内容长度限制 | 单篇 markdown 文本最大 5 MiB（防止极端长输入耗光内存）；超出时构图阶段抛 `ContentGraphException`。该上限可在 `Bocchi:Generator:MaxBodyBytes` 覆盖。 |

## 6. 风险与回避

| 风险 | 回避 |
| --- | --- |
| 在 Theme 真正落地前提前实现 artifact 级增量、把不变量塞进 Theme 协议 | 本文件 §3.12 明确把细粒度增量延后；M3 仅做整次指纹判定。 |
| `output/public/` 与内容空间混淆 | 路径全部经由 `WorkspaceLayout`；`output/public/` 与 `.bocchi/input/` 全部位于内容空间根**之外**；`FileSystemBuildSink` 写入前一律 `Path.GetFullPath` 并断言前缀。 |
| 实时模式与本地静态模式行为发散 | 共用 Pipeline + Stage，唯一差异在 Sink；通过测试 `PipelineEndToEndTests` 与 `PreviewEndpointTests` 双向 pin 住一致性。 |
| Theme 进程长时间未结束 | `ThemeRunner` 强制 `CancellationToken` + 超时（默认 10 分钟）；进程被 kill 时记 `ThemeRunnerException`，不污染 sink。 |
| 在没有任何 Theme 时构建报错 | `RunThemeBuildStage` 把"无可用 Theme"显式建模为 `Skipped`，Pipeline 仍然成功。 |
| Pagefind 后期反向施压要求改架构 | 决策已记录于 §3.10；若日后启用，作为可选 sink 出现，不替换默认。 |
| JSON 输出顺序不稳定 → Git diff 噪声 | DTO 使用稳定字段顺序；数组（Posts / Pages / Works / Notes / Friends）使用 §3.3 规定的稳定排序键；CamelCase + 缩进；UTF-8 无 BOM。 |
| 大型媒体复制阻塞构建 | `CopyMediaStage` 使用 `FileStream` 异步流式复制 + `CancellationToken`；mtime 一致 + Sha256 一致时跳过实际拷贝（M3 起就为媒体走"内容地址化"复用）。 |
| LibGit2Sharp / SQLite 在 CI 上的本机依赖问题 | 复用 M2 已验证的 NuGet 包；无新增本机依赖。 |
| 测试在不同操作系统上路径分隔符不一致 | 所有比较走"以 `/` 归一化的相对路径"；新增 `PathNormalization` 测试断言。 |

## 7. 移交给下一阶段（M4 / M5 / M6）的事项

### 7.1 给 M4 Home Server Dashboard

- `/build` 页面目前是最小同步页面；M4 接入实时 SignalR 日志流、阶段进度条、artifact 树等富交互。
- "修正性回写到内容空间"的 UI 入口（§3.1 第三条）：M3 仅约定 API 形状（"扫描发现 → Dashboard 提示 → 用户点击 Apply Fix"），M4 落地。
- Theme 配置 UI（schema → 表单）所读的 schema 文件位置 / 默认值合并策略在 M3 中通过 `theme-config.json` 信封固定下来；M4 只填表，不改协议。

### 7.2 给 M5 Default SvelteKit Theme

- Theme 消费契约：从 `.bocchi/input/*.json` 读取；环境变量遵循 §3.8；输出到 `theme.json.outputDir`（默认 `build/`），Home Server 在 `RunThemeBuildStage` 之后会把 Theme 的输出目录原样合入 `output/public/`（M5 提供 `CopyThemeOutputStage` 即可纳入 Pipeline，不需要改 M3 的 Stage 顺序）。
- Theme 正文表达三选一（`markdown` / `html` / `excerpt`）；代码高亮 / 图片放大 / 数学公式 / 视音频 / 表格等全部由 Theme 决定如何处理。
- 配置化"功能性页面"由 Theme 自行定义；其数据来源是 `theme-config.json`，Home Server 不需要事先知道页面结构。

### 7.3 给 M6 Feeds / Search / Publish

- 静态搜索索引（自研 JSON）：从 `.bocchi/input/posts.json` 等读，无需重扫内容空间。
- 发布目标（Local / Cloudflare Pages）：从 `output/public/` 出发；可读 `.bocchi-manifest.json` 做对账与增量上传。
- 细粒度增量构建（artifact 级依赖图）：在 §3.9 表结构基础上新增 `ArtifactDependencies` 表与 `SourceFingerprintRevisionId`；具体设计与 Theme 协议一同确定。
- Git 远程接入（push/pull / GitHub）：与发布管线一起处理。
