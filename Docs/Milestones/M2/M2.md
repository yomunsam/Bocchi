# M2 Content Workspace

本文件是 M2 阶段的"作战图"。它把 `Docs/Milestones.md` 中 M2 的目标拆成可勾选的最小项，并把 M2 引入的关键设计决策（特别是"内容空间 / Bocchi 系统空间"切分、年份一级分类、内容空间作为 Git 仓库、源工程 vs 构建产物的边界）固化下来，作为后续阶段不可回退的护栏。

> 上下文恢复入口：阅读顺序为 `Docs/Architecture.md` → `Docs/Milestones.md` → `Docs/Milestones/M1/M1.md` → 本文件。
>
> M2 的任何实现都必须与这里的"关键决策"一致；如需调整，先改本文件再改代码。

## 1. 目标

让 Bocchi 真正"读懂"本地内容，并把"长期可控、可一键带走的创作内容"和"Bocchi 自己的系统状态"在目录、抽象、依赖三层都做出干净切分。

具体落点：

- 一个真实可用的 `IWorkspace` 加载/初始化实现，能从一个目录还原出完整的工作区视图。
- 一个"内容空间"（Content Space）的正式目录约定：用户的 Blog、Page、作品、短文、友链、站点设置等纯创作内容都在这里，可以独立打包、迁移、用 Git 管理。
- 一个"Bocchi 系统空间"：Theme、Theme 配置、SQLite 状态、日志、缓存、构建产物等，全部留在 `.bocchi/`、`themes/`、`output/` 等系统目录里，与内容空间互不污染。
- Markdown + YAML frontmatter 的解析管线，能把 MVP 七类内容里的六类（Post、Page、Work、Note、FriendLink、SiteSettings；Photo 仅占布局位）全部加载为强类型对象。
- 一份 SQLite 管理状态：记录扫描快照、文件 hash、解析错误、媒体引用、最近一次扫描结果，**不复制内容正文**。
- 内容校验与可追溯的错误报告：错误必须能定位到文件 + 字段。
- 通过 `LibGit2Sharp` 把内容空间识别为 Git 工作区：能 init、能读 status、能在内网后台触发提交。远程接入（GitHub、push/pull、auth）作为 M2 的*非目标*显式延后到后续里程碑。
- Home Server 后台首页消费扫描结果，展示"工作区状态"与"内容空间 Git 状态"。
- 从 M1 遗留的"Serilog 文件 sink 暂落到运行目录 `logs/`"切换到 `<workspace>/.bocchi/logs/`。

## 2. 范围与非目标

在范围内：

- Workspace / Content Space 目录约定与初始化。
- Markdown / frontmatter 解析（决定使用 YAML，写入决策记录）。
- 七类内容的目录与文件结构（Photo 仅占位）。
- SQLite 状态库的 schema 与基础 CRUD（schema 版本字段，简单显式迁移）。
- 内容扫描与校验、错误结构化输出。
- Git 仓库识别 + init + status + 提交（本地）。
- 后台"工作区"页面：路径、初始化按钮、扫描按钮、错误列表、Git 状态。
- 自动写出"内容空间使用说明"`README.md` 与一份合理的 `.gitignore`。
- 公共抽象 + DI 注册（`AddBocchiWorkspace(...)` 等扩展方法）。
- 单元测试 + 端到端集成测试（含"样例工作区"fixture）。
- 在 Architecture / Milestones 总文档中固化决策。

非目标（移交给后续里程碑）：

- M3：标准化内容图、Theme 输入数据写入、媒体衍生品（webp / 缩略图）、构建 manifest、CLI / 后台触发完整构建。
- M4：完整后台管理界面（内容列表、富表单、Theme 配置 UI 表单、构建日志页）。M2 只交付一个最小的"工作区状态"页用于回路验证。

- M6：RSS / Sitemap / 搜索 / 发布。
- Photo 的真实扫描与照片墙体验。
- Git 远程：远程仓库管理、push/pull、凭据存储、与 GitHub 的直接连接。这些会在 M6（发布）或之后专门处理；M2 仅保证"内容空间可以是 Git 仓库"这一架构事实成立。
- 增量构建、依赖图、缓存命中等性能优化。M2 的扫描是"全量 + hash 比对跳过解析"水平。

## 3. 关键决策

这些决策在 M2 落定，后续不再轻易回退；如需调整，必须先改本节并同步 `Docs/Architecture.md`、`Docs/Milestones.md`。

### 3.1 内容空间 vs Bocchi 系统空间（最核心）

| 维度 | 内容空间（Content Space） | Bocchi 系统空间 |
| --- | --- | --- |
| 默认位置 | `<workspace>/content/` | `<workspace>/themes/`、`<workspace>/.bocchi/`、`<workspace>/output/` |
| 归属 | 用户自己的创作资产 | Bocchi 这个程序的实现细节 |
| 存活周期 | 和用户一生 / 写作活动一样长 | 和 Bocchi 这个项目同寿，可被替换 |
| 是否可独立 Git | **是**，本身就是一个完整的、可独立提交/迁移的源工程 | 否；不应被 commit 到内容仓库 |
| 是否包含构建产物 | **否**。仅原始 Markdown / 原始媒体 / 文本配置 | 是；衍生媒体（webp、缩略图、HTML、搜索索引）只允许在这里 |
| 是否包含 Theme 专有配置 | **否**。Theme 自身的 `config-schema.json` 默认值 + Theme 配置实例 都在系统空间 | 是；`.bocchi/theme-config/` |
| 是否包含 Theme 实现 | 否 | 是；`themes/` |

派生宗旨："Bocchi 这个程序未来可能会过时、被遗弃、被重构；但内容空间作为一个独立的源工程，应当能一键打包带走，不带任何 Bocchi 痕迹"。M2 的所有 IO 路径都由 `IWorkspaceLayout` / `IContentSpaceLayout` 提供，禁止任何模块直接拼路径绕过抽象。

### 3.2 内容空间目录约定

```text
content/                                      <-- Content Space 根（可以是 Git 仓库）
  README.md                                   <-- 自动生成，说明本目录是"内容源工程"
  .gitignore                                  <-- 自动生成，排除 macOS / 编辑器临时文件
  posts/
    2026/                                     <-- 年份一级分类（强约束）
      hello-bocchi/                           <-- 文章 slug 即文件夹名
        index.md                              <-- frontmatter + Markdown 正文
        assets/
          cover.jpg
          figure-1.png
  pages/
    about/                                    <-- 独立页面：无年份层；slug 即文件夹名
      index.md
      assets/
  works/
    2024/                                     <-- 作品同样按年份归档
      my-game/
        index.md
        assets/
  notes/
    2026/                                     <-- 短文按年份分目录
      2026-05-13-2030-coffee.md               <-- 单文件即一条短文，文件名含时间戳 + 简短 slug
  friends/
    friends.yaml                              <-- 友链：单文件 YAML 列表
  photos/                                     <-- M2 仅占位，内部约定推迟到照片墙正式开发
  site/
    site.yaml                                 <-- 站点设置
    navigation.yaml                           <-- 导航
```

强约束：

- `posts/`、`works/`、`notes/`、`photos/` 一律使用 **年份目录** 作为一级分类；年份必须是四位数字（正则 `^\d{4}$`）。
- Post / Work 单篇为"目录形式"：`<kind>/<year>/<slug>/index.md` + `assets/`；`assets/` 仅放该篇引用的原始媒体（jpg/png/svg/mp4/...）。
- 短文为"单文件形式"：`notes/<year>/<filename>.md`；不强制目录化（短文体量小，单文件更利于浏览）。
- 单篇引用的媒体路径在 frontmatter 中以 **相对当前文件** 写：`cover: assets/cover.jpg`。生成器在 M3 才把它改写为站点路径。
- Pages 不按年份分类（独立页面是"地点"而不是"事件"），目录形式：`pages/<slug>/index.md`。

### 3.3 frontmatter 与 Markdown

| 项 | 决策 | 理由 |
| --- | --- | --- |
| frontmatter 格式 | **YAML**，使用 `---` 分隔 | 生态最广，Markdig 与 YamlDotNet 原生支持；TOML 不增量价值 |
| Markdown 引擎 | `Markdig` | .NET 事实标准；对 frontmatter / 自定义扩展友好 |
| YAML 引擎 | `YamlDotNet` | .NET 事实标准；社区活跃 |
| frontmatter 字段命名 | camelCase（与 `Docs/Architecture.md §4` 示例一致） | 与 JS / Theme 输入侧一致，减少映射 |
| frontmatter 必需字段缺失 | 计入校验错误，**不抛异常**；扫描继续 | 错误聚合 > 早早 fail |
| Slug 来源 | 文件夹名（Post/Work/Page）或 frontmatter 显式覆盖 | 文件夹本身就是 slug 的事实来源；frontmatter 仅在迁移历史时覆盖 |
| 草稿态 | `status: draft`（默认）/ `published` | 与 ContentModel `ContentStatus` 一致 |
| 时区 | 时间字段统一以 `DateTimeOffset` 解析；缺少时区视为站点配置的 `timeZone` | 避免 "naive datetime" 歧义 |

> 这同时关闭 `Docs/Milestones.md` 待决问题中的 "Markdown frontmatter 采用 YAML、TOML 还是兼容多格式"：仅采用 YAML，未来需要扩展时再走 RFC。

### 3.4 短文存储格式

| 项 | 决策 | 理由 |
| --- | --- | --- |
| 短文存储 | **Markdown 文件**，单文件即一条短文 | 与"内容空间可独立带走"一致；JSON Lines 不便于直接编辑、不亲和 Git diff；SQLite 完全违背"内容事实在文件"原则 |
| 文件名 | `YYYY-MM-DD-HHMM-<short-slug>.md`，频率高时允许加秒 | 自然时间排序、Git diff 友好 |
| frontmatter 字段 | `status`、`publishedAt`（可省略，回退到文件名解析）、`tags`、`media` | 对应 `Note` 记录 |
| 正文 | 即 Markdown 正文（短文的 `text` 字段直接来自正文，不在 frontmatter 中重复） | 避免双重事实来源 |

> 关闭 `Docs/Milestones.md` 待决问题："短文的初始存储格式使用 Markdown、JSON Lines 还是 SQLite + 导出" → Markdown。

### 3.5 SQLite 状态库

| 项 | 决策 |
| --- | --- |
| 数据库引擎 | `Microsoft.Data.Sqlite`（轻量，无 EF Core 仪式） |
| 数据库文件 | `<workspace>/.bocchi/bocchi.sqlite` |
| schema 版本 | `PRAGMA user_version`，由 `SchemaMigrator` 显式管理 |
| M2 表 | `Files`（路径、kind、hash、最后修改时间、最后扫描时间）<br>`ContentItems`（kind、id、slug、title、status、publishedAt、updatedAt、source file id、frontmatter JSON）<br>`MediaReferences`（owner content id、relative path、resolved absolute path、是否存在）<br>`ContentErrors`（file id、字段、严重级、消息、扫描批次）<br>`ScanRuns`（开始/结束时间、文件总数、错误数、git head sha 快照） |
| 写入策略 | 全部走事务；扫描总是"先开 ScanRun → upsert Files / ContentItems → 收尾或回滚" |
| 内容正文 | **不入库**；只入 hash + 解析后的 frontmatter（用于快速列举） |
| 并发 | 全程串行；M2 不引入并发扫描，但所有 API 接收 `CancellationToken` |
| 连接管理 | 单一 `SqliteConnectionFactory`，Singleton；连接按调用打开/关闭，避免长连接问题 |
| 路径 | 全部以 *相对内容空间根* 形式入库；显示时再拼绝对路径，迁移内容空间时不破坏数据 |

### 3.6 Git 集成（内容空间侧）

| 项 | 决策 |
| --- | --- |
| 库 | `LibGit2Sharp` 0.31.0 |
| 作用域 | 仅作用于 **内容空间根**，不操作 Bocchi 系统空间或仓库本身 |
| M2 提供能力 | `IsGitRepository`、`InitRepository`、`GetStatus`（dirty 文件计数 + 最新提交）、`Commit(message, author)` |
| M2 不提供 | `Remote` / `Push` / `Pull` / GitHub OAuth / 凭据持久化 |
| 默认 author | 由 site settings `author.name` + `author.email` 提供，可被显式参数覆盖 |
| 失败策略 | LibGit2Sharp 异常被包装为 `ContentGitException`，不上抛原始本机异常 |
| 显示 | 在 Home Server 后台展示 "未初始化 / 干净 / N 个未提交修改" + 最新提交摘要 |

> 长远目标（连接 GitHub、用 Git 做版本管理）：M2 完成"基础设施 + 本地能力"，**没有**完成"远程接入"。这是有意的延后，避免在没有发布管线（M6）的情况下提前引入凭据/Webhook 等复杂度。2026-05-14 复查官方仓库 / NuGet 后确认：`LibGit2Sharp` 0.31.0 是当前稳定版本，包目标包含 `net8.0`，NuGet 计算兼容 `net10.0`；Bocchi 当前仅使用本地 init/status/commitAll，继续使用它是合适的。`Docs/Milestones.md` 待决问题中保留 "Git 远程接入与发布管线的衔接" 一项，明确推迟到 M6。

### 3.7 源工程 vs 构建产物边界

- 内容空间禁止出现 `*.webp`、`*.thumb.jpg`、`*.html`、`pagefind/` 等任何派生产物。
- M3 的衍生媒体目标目录固定为 `<workspace>/.bocchi/cache/derivatives/`；M2 不实现任何衍生处理，只把"边界"写进文档与扫描器（扫描器遇到内容空间内带可疑派生扩展名的文件时，发出 warning 级 `ContentError`）。
- 自动生成的 `.gitignore` 默认禁止 `assets/` 中出现已知派生扩展名；用户可显式覆盖。

### 3.8 工程层决策

| 主题 | 决策 |
| --- | --- |
| Markdown / YAML 解析放置 | `Bocchi.Workspace`，不下沉到 `Bocchi.ContentModel`（保持后者零外部依赖） |
| SQLite 实现放置 | `Bocchi.Workspace`，子命名空间 `Bocchi.Workspace.State`；不另开新项目（M2 体量不值得） |
| Git 实现放置 | `Bocchi.Workspace`，子命名空间 `Bocchi.Workspace.Git`；接口在 `Bocchi.Workspace`，实现内部 |
| 公共 DI 注册 | `Bocchi.Workspace.DependencyInjection.WorkspaceServiceCollectionExtensions.AddBocchiWorkspace(IConfiguration)`；Home Server 通过此扩展接入 |
| 配置节 | `appsettings.json` 中 `"Bocchi": { "WorkspaceRoot": "..." }`；未配置时回退到 `<contentRoot>/workspace/` 并自动初始化 |
| Serilog 文件 sink | M2 切换到 `<workspace>/.bocchi/logs/bocchi-.log`，按天滚动 14 天 |
| 异常类型 | 每个失败模式都有专属异常：`WorkspaceInitializationException`、`ContentParseException`、`ContentStateException`、`ContentGitException` |
| 时间源 | 一律 `TimeProvider`，禁止 `DateTime.UtcNow` 直接使用 |
| 字符串 | 文件路径相关使用 `Path.Combine` / `Path.GetRelativePath`；YAML 路径用 `'/'` 归一化 |
| 异步 | IO 操作走 async；CPU 密集（YAML/MD 解析）保持同步即可 |

## 6. 风险与回避

| 风险 | 回避 |
| --- | --- |
| LibGit2Sharp 在 CI / Windows 环境出现本机依赖或文件句柄问题 | M2 仅依赖 LibGit2Sharp 的标准 NuGet 包；不依赖系统级 libgit2；CI 走 `dotnet test` 即可。Windows 下 `.git/objects` 临时文件清理可能短暂被占用，测试辅助器采用重试 + 只读属性清理，不把清理时机误判为业务失败。 |
| SQLite 在 net10.0 + 集中包管理下与隐式包冲突 | `Microsoft.Data.Sqlite` 显式登记 PackageVersion 即可；不引入 EF Core，避免 Provider 链 |
| Markdig 与 YAML frontmatter 扩展冲突 | 使用官方 `UseYamlFrontMatter` 扩展，不自己重写 frontmatter 切分；frontmatter 提取仍走自家 `FrontmatterParser`（解析 YAML 用 YamlDotNet），Markdig 只负责正文渲染 |
| 内容空间作为 Git 仓库时与 Bocchi 系统状态目录混淆 | `.bocchi/`、`themes/`、`output/` 全部位于 **内容空间根之外**；`LibGit2ContentRepository` 的工作树严格锁定到 `ContentSpaceLayout.Root`，不会误把 `.bocchi/` 纳入索引 |
| 时间字段时区缺失导致排序漂移 | 解析时强制 `DateTimeOffset`；缺时区时回退到 site.yaml `timeZone`；缺 site.yaml 时回退到系统当地，但记一条 warning |
| 把 M3 的"标准化内容图 / Theme 输入数据"逻辑提前塞进 M2 | M2 的所有 ContentScanner 输出停留在"内存中的强类型 + SQLite 记录"，不写出任何 `.bocchi/input/*.json`；超出范围的 PR 必须延后 |
| 在 M2 仓促引入"Git 远程"导致凭据 / 权限设计提前 | 本文件 §3.6 明确把远程相关延后；任何 PR 想加 push/pull/auth 必须先改 §3.6 |

## 7. 移交给下一阶段（M3）的事项

- 标准化内容图：把 M2 输出的 `PostDocument` / `PageDocument` / ... 合并、解 reference、生成站点视角的导航与索引。
- Theme 输入数据写入：`<workspace>/.bocchi/input/posts.json` 等。媒体路径在此阶段从"相对源文件"改写为"站点路径"。
- 媒体衍生品：在 `<workspace>/.bocchi/cache/derivatives/` 中生成 webp / 缩略图（M3 真正落地"源工程 vs 构建产物"边界）。
- 首次构建：CLI 或后台按钮触发 Full Build，产出 `<workspace>/output/public/`。
- M2 的 `ContentScanner` 在 M3 中应当能被 Generator 复用，而非重新扫一遍。
- Git 远程：连同 M6 发布管线一起设计，是否把"提交并 push"作为发布流水线的一步。
