# M6 内容资产管理专项

本文记录文章编辑器插图、上传、粘贴、附件与内容级资产面板的专项规划。它是 M6 之后补上的内容创作体验专项，不改变 M6 localization 主线已完成的结论，也不属于 M7 Feeds / Search / Publish 主线。

## 1. 背景

当前 Bocchi 已经具备内容资产的底层事实：

- workspace 约定 Post / Work 使用目录型内容：`index.md` + `assets/`。
- Page 使用目录型内容：`pages/<slug>/index.md`，也可以自然携带 `assets/`。
- M6 多语言内容版本使用同一 localization group，同一目录下的 `index.md`、`index.en-US.md` 等语言 variant 应共享 `assets/`。
- Generator 已经能从 Markdown / frontmatter 收集媒体引用，把内容资产复制到 `output/public/media/...`，并在 theme input 中输出 media 信息。
- `EditorDraftService` 已经有临时草稿目录和 `assets/`，并提供把上传或粘贴资产写入 draft assets 的基础能力。
- 当前 Markdown editor 只有图片按钮和 paste/drop 文本占位逻辑：它会插入 `assets/<name>`，但没有真正上传文件，也没有资产面板。

因此本专项不应另起一套全站媒体库。第一阶段应补齐“当前内容组自己的 assets 管理”和“编辑器插入资产”的闭环。

## 2. 目标

1. 文章、页面、作品、短文都能在编辑器中插入图片和附件。
2. 粘贴板图片、拖拽文件、文件选择上传都能写入当前内容组的 `assets/`，并把 Markdown 引用插入光标位置。
3. 编辑器提供当前内容组的资产管理面板，能查看、插入、复制路径、删除未引用资产。
4. 未保存的新内容先写入 editor draft assets；第一次保存到 workspace 时，assets 随内容一起迁移。
5. 已保存内容直接写入内容目录的 `assets/`。
6. 多语言 variant 共享同一内容组资产目录，不为每个语言文件拆一套资产。
7. 保持 workspace 是用户内容事实来源；资产元数据第一阶段不进入数据库。

## 3. 非目标

- 不做全站媒体库或跨文章复用资产。
- 不做 CDN、对象存储、外部媒体托管。
- 不做图片裁剪、压缩、缩略图、webp/avif 派生或响应式图片生成。
- 不做远程 URL 自动下载。
- 不做富文本式图片布局编辑器。
- 不做复杂附件权限系统。对外静态站点中的内容资产默认随站点发布。

## 4. 核心决策

### 4.1 资产归属到内容组

资产归属单位是内容组，而不是单个语言 variant。

```text
posts/2026/hello/
  index.md
  index.en-US.md
  assets/
    cover.jpg
    chart-en.png

pages/about/
  index.md
  index.en-US.md
  assets/
    avatar.jpg
```

理由：

- 同一内容的多个语言版本通常共享封面、截图和附件。
- 当前 M6 localization group 已经采用同目录 sibling 文件模型，`assets/` 自然属于这个目录。
- Generator 已经按内容目录解析相对媒体路径，继续使用 `assets/<file>` 最稳。

### 4.2 Note 使用目录型存储与短稳定 id

Note / 短文采用目录型存储。单文件 Note 是早期轻量文本模型，不再作为目标形态；后续实现不需要兼容旧单文件 Note。

workspace 目录负责按时间分桶，公开 URL 只暴露短稳定 id：

```text
notes/2026/0529/1432-k7p9xq2m/
  index.md
  assets/
    image.png

public URL:
/notes/k7p9xq2m/
```

目录规则：

```text
notes/{yyyy}/{MMdd}/{HHmm}-{id}/index.md
```

`id` 写入 frontmatter，作为 Note 的永久公开标识：

```yaml
---
id: k7p9xq2m
status: published
createdAt: 2026-05-29T14:32:00+08:00
updatedAt: 2026-05-29T14:32:00+08:00
---
```

规则：

- `id` 为 8 位小写字母与数字，创建时生成，永久不变。
- UI 不允许用户编辑 `id`。
- 目录名中的 `id` 必须与 frontmatter `id` 一致；不一致视为内容错误。
- 同一分钟创建多条 Note 时不需要时间后缀冲突处理，`id` 负责唯一性；如果目录已存在则重新生成 `id`。
- `yyyy/MMdd/HHmm` 只用于 workspace 浏览和 Git diff 组织，不进入公开 URL。
- SQLite 扫描投影如有需要，可以建立 `noteId -> sourcePath` 映射。
- Note 不参与 M6 内容源文件层多语言 variant 模型，不生成 `index.en-US.md`。未来翻译属于前台或 Cloud Server 展示/缓存能力。

Note media prefix 建议与公开 id 对齐：

```text
/media/notes/{id}/{file}
```

这样文件在 workspace 内移动时，只要 frontmatter `id` 不变，Note URL 和媒体 URL 都保持稳定。

短文目录化需要单独小切片处理：

- Workspace 初始化文档和 Architecture 更新。
- Note loader 只支持目录型 Note，并校验 `id` 格式、重复和目录一致性。
- Generator Note URL 输出为 `/notes/{id}/`。
- Generator media prefix 输出为 `/media/notes/{id}/...`。
- HomeServer Note 创建和编辑入口切换到目录型。

### 4.3 远程 URL 第一版只插入，不导入

远程 URL 可以作为 Markdown 链接或图片引用插入，但第一版不自动下载到本地 assets。

理由：

- 服务端下载远程 URL 会引入 SSRF、内网地址访问、重定向、超时、文件大小和 Content-Type 校验等问题。
- “插入远程 URL”与“导入远程文件”是两个不同产品动作，应分开实现。

后续如做导入，需要专门的安全边界：禁止私有网段、限制大小、限制重定向、限制 MIME、超时和错误展示。

### 4.4 资产面板不是媒体库

资产面板只管理当前内容组的 `assets/`。

它不负责跨内容搜索、复用、标签、图库或媒体中心能力。未来如果需要全站媒体库，应另立专项，不能把第一版资产面板扩成 DAM。

## 5. 推荐 UI 形态

### 5.1 Markdown toolbar

现有图片按钮保留，但行为升级：

- 点击图片按钮：打开文件选择，选择图片后上传并插入 `![alt](assets/file.ext)`。
- 选择普通附件：插入 `[file.ext](assets/file.ext)`。
- 粘贴图片：上传到 assets，并插入图片 Markdown。
- 拖拽文件到编辑区：上传到 assets，并插入图片或附件 Markdown。

按钮仍以图标为主，tooltip / aria-label 使用 Dashboard i18n。

### 5.2 编辑器资产面板

在 ContentEditor 右侧新增一个 Assets panel，与状态、语言版本 panel 同级。

建议分组：

| 分组 | 含义 |
| --- | --- |
| Referenced | Markdown / frontmatter 正在引用的本地 `assets/...` 文件。 |
| Unused | `assets/` 中存在但当前内容没有引用的文件。 |
| External | Markdown 中的远程 URL 图片或附件。第一版只展示，不管理。 |

每个本地资产项显示：

- 图片缩略图或文件类型 icon。
- 文件名。
- 文件大小。
- 引用状态。
- 操作：插入、复制路径、删除。

删除策略：

- 未引用资产可以直接删除或二次确认删除。
- 已引用资产默认不允许直接删除，或必须显示明确确认，提示会产生 broken reference。

## 6. 存储与服务边界

### 6.1 Draft 内容

新建内容在未落入 workspace 前使用：

```text
data/state/editor-drafts/<draft-id>/
  content.md
  session.json
  assets/
```

上传、粘贴、拖拽资产先写入 draft `assets/`。发布或保存为正式内容时，现有 `ContentEditingService.CreateFromDraftAsync` 会把 draft assets 移到最终内容目录。

### 6.2 已保存内容

已保存 Post / Page / Work 直接写入内容目录的 `assets/`。

如果保存时 slug 导致内容目录移动，现有目录移动逻辑会带走 `assets/`。

### 6.3 Note 内容

Note 使用目录型内容：

```text
notes/{yyyy}/{MMdd}/{HHmm}-{id}/
  index.md
  assets/
```

`id` 是 Note 的公开稳定标识，必须写入 `index.md` frontmatter。Note 的公开 URL 和后续动态服务器逻辑都使用 `id`，不使用 workspace 日期目录。不要为单文件 Note 开旁路资产目录。

## 7. 工作包

### CA-01 Asset Service

目标：建立 HomeServer 内容资产服务，封装文件系统操作。

职责：

- 解析当前编辑对象的资产目录：draft 或 workspace 内容目录。
- 枚举 `assets/` 文件。
- 写入上传文件并生成安全唯一文件名。
- 删除资产。
- 根据当前 Markdown / frontmatter 区分 referenced / unused。
- 返回 UI 需要的 asset view model。

验收：

- 路径穿越被拒绝。
- 文件名被规范化。
- 同名文件自动加后缀。
- draft assets 和 workspace assets 都可用。
- 没有数据库依赖。

### CA-02 Markdown Editor Upload Bridge

目标：让 CodeMirror paste/drop/file picker 能把浏览器 File 交给 Blazor，再插入返回的 Markdown。

职责：

- JS 捕获 paste/drop 的 File。
- 调用 .NET 上传方法，而不是只插入占位路径。
- 上传成功后在光标处插入 `![alt](assets/file)` 或 `[file](assets/file)`。
- 失败时给出 Dashboard i18n 错误提示。

验收：

- 粘贴图片能在 draft 中生成真实文件。
- 拖拽图片能生成真实文件。
- 文件选择上传能生成真实文件。
- 预览 HTML 能立即显示本地图片。

### CA-03 ContentEditor Assets Panel

目标：在编辑器右侧加入当前内容组资产面板。

职责：

- 显示 referenced / unused / external。
- 支持插入现有资产。
- 支持复制相对路径。
- 支持删除未引用资产。
- 对已引用删除给出明确保护。

验收：

- 单语言、无资产内容不增加明显负担。
- 有资产内容能清楚看到当前文件是否被正文引用。
- UI 文案全部走 Dashboard i18n。

### CA-04 Note Directory Model

目标：把短文固定为目录型内容，并用 8 位短 id 作为公开 URL 与后续动态能力的稳定锚点。

职责：

- 定义 `notes/{yyyy}/{MMdd}/{HHmm}-{id}/index.md` workspace 结构。
- 创建时生成 8 位小写字母数字 `id`，写入 frontmatter，并用于目录名。
- Note loader 只支持目录型 Note。
- 校验 Note `id` 缺失、格式非法、重复、目录 id 与 frontmatter id 不一致。
- Note 创建入口写入目录型 Note。
- Generator Note URL 输出为 `/notes/{id}/`。
- Generator media path 输出为 `/media/notes/{id}/...`。
- 文档更新 Architecture / Milestones。

验收：

- 新 Note 可以插入图片。
- Note URL 不暴露日期目录。
- Generator 输出 note media。
- 现有 note tests 覆盖目录型。

### CA-05 Attachment Types

目标：在图片之外支持普通附件。

第一版建议允许：

- 图片：jpg、jpeg、png、gif、webp、avif。
- 文档附件：pdf、txt、md。
- 压缩包是否允许需要单独确认。

SVG 不建议第一版通过编辑器上传。虽然项目已有静态输出 MIME 支持，但 SVG 作为用户上传内容存在脚本和外链风险；后续可以通过“受信任用户上传”或 sanitizer 决策放开。

验收：

- 图片插入为 image markdown。
- 非图片附件插入为 link markdown。
- MIME / extension 不符合预期时拒绝。

## 8. 测试建议

- `EditorDraftService`：上传资产、文件名规范化、同名冲突、draft 删除。
- `ContentAssetService`：枚举 referenced / unused、路径穿越拒绝、删除策略。
- `ContentEditingService`：draft assets 发布到 workspace、内容目录移动后 assets 保留。
- `ContentScanner`：缺失媒体和孤儿 assets 继续 warning。
- `Generator`：Post / Page / Work / Note 目录型资产复制到 `/media/...`。
- `HomeServer`：ContentEditor 上传 endpoint / component 行为。
- 浏览器 smoke：粘贴图片、拖拽图片、上传附件、保存后预览。

## 9. 开放问题

1. 第一版附件是否允许 PDF 之外的二进制文件。
2. SVG 是否彻底禁止，还是仅禁止未受信任上传。
3. 资产删除是否允许删除已引用文件，还是必须先移除引用。
4. 远程 URL 是否只插入，还是提供明确的“导入到本地 assets”后续任务。
5. 是否需要在 assets panel 中支持“设为封面图”。

## 10. 推荐第一轮执行顺序

1. 实现 Asset Service，并覆盖 draft/workspace 两种根。
2. 实现 Note 目录型与短稳定 id。
3. 串起 Markdown editor paste/drop/upload 到 draft assets。
4. 加 ContentEditor Assets panel 的最小可用版本。
5. 处理已保存内容 assets。
6. 再进入附件类型扩展。

第一轮成功标准：新建文章时粘贴一张图片，文件真实进入 draft `assets/`，正文插入相对路径，预览可见；保存为正式内容后，图片随内容进入 workspace `assets/`，构建后出现在静态站点。
