# M6 Localization and Content i18n

本文件是 M6 阶段的“作战图”。它把 Dashboard 自身 i18n、站点语言设置、前台 Theme 本地化约定、内容多语言版本、URL/SEO 输出和后续翻译能力边界固定下来，确保本次讨论丢失后也能从文档恢复完整设计。

> 上下文恢复入口：阅读顺序为 `Docs/Architecture.md` → `Docs/Milestones.md` → `Docs/Milestones/M4/M4.md` → `Docs/Milestones/M5/M5.md` → 本文件。
>
> M6 以后，Bocchi 开始把“多语言站点”作为产品能力处理。Dashboard 自身的语言切换仍是后台 UI 偏好；前台站点语言、内容语言版本和 Theme 文案覆盖是站点能力，两者必须分开。
>
> Frontend Menu v1 与 Theme Page Contract 的详细实施记录见 `Docs/Milestones/M6/Menu.md`。
>
> Theme 作者开发、Theme Library、Zip Package、Dev Link 和 process runner trust 的专项记录见 `Docs/Milestones/M6/Theme开发专项.md`。该专项是 M6 期间为 Theme 生态补上的并行工作，不改变 M6 本地化主线的验收边界。

## 1. 目标

让 Bocchi 在不打断普通写作流程的前提下，具备清晰、可扩展的本地化能力：

1. Dashboard 自身支持 `zh-CN` 和 `en-US` 两种 UI 语言，并以可扩展到更多语言的选择控件呈现。
2. Dashboard 提供站点本地化设置：主要语言、站点启用语言、自定义语言、通用前台文案覆盖。
3. Theme Contract 能把语言配置、语言元数据、文案覆盖和内容语言版本交给前台 Theme。
4. 默认 Theme `default-static` / `Bocchi Mono` 给出基础示范，首批兼容 `en-US`、`zh-CN`、`zh-TW`、`ja-JP`。
5. Post / Page / Work 支持同一内容的多个语言版本，并能区分“原生语言版本”和“翻译版本”。
6. 静态输出采用稳定的语言路径、`hreflang` 和 canonical 元数据，保证分享、构建、预览和 SEO 行为可预测。
7. 翻译能力先以手动编辑闭环为基础；Translation Provider 保留为后续增强入口，只有出现真实、可验证的 provider 时再落地抽象和 UI。

## 2. 核心术语

| 术语 | 含义 |
| --- | --- |
| Dashboard UI language | Admin 操作 Dashboard 时使用的界面语言，是后台 UI 偏好，不传给 Theme。 |
| Site primary language | 站点主要语言，用于默认内容语言、无前缀 URL、默认 `<html lang>` 和 fallback 起点。 |
| Site enabled languages | 站点启用语言列表，必须包含主要语言。它表示站点愿意声明和管理的语言集合。 |
| Language record | 一个语言的描述，包含语言代码、当地语言名称和英文名称。 |
| Common i18n key | Bocchi 对前台 Theme 约定的一组通用文案 key，例如 `menu.home`、`common.readMore`。 |
| Theme i18n key | 某个 Theme 在 manifest 中声明的私有文案 key，只对该 Theme 有意义。 |
| Localization group | 同一篇文章、页面或作品在不同语言下的逻辑分组。 |
| Content variant | Localization group 下的一个具体语言版本。 |
| Native variant | 用户直接写作的语言版本，不声明自己翻译自其他版本。 |
| Translation variant | 明确记录自己翻译自同组某个 Native variant 或其他 variant 的语言版本。 |

## 3. 设计边界

- 普通单语言写作流程不出现强制语言步骤。用户新建内容、写 Markdown、保存或发布时，体验应与 M5 单语言站点一致。
- 内容没有显式 `language` 时，系统按 Site primary language 看待它。
- Dashboard UI language 与 Site primary language 是两个独立设置。Admin 可以用中文后台管理英文站点，也可以反过来。
- Dashboard 的语言切换必须支持大于 2 种语言，不使用二态 Toggle。
- Dashboard、Home Server 和默认 Theme 都不显示语言图标或国旗。Theme 可以自行用语言代码关联图标，但这不进入 Bocchi 通用约定。
- 语言代码建议使用 BCP 47 tag，例如 `en-US`、`zh-CN`、`zh-TW`、`ja-JP`。用户自定义语言代码时，Dashboard 给出建议但不强行限制。
- 前台文案覆盖值第一版只支持 plain text。Theme 渲染时按普通文本处理，不把覆盖值当 HTML 或 Markdown。
- M6 的内容多语言模型覆盖 Post / Page / Work。Note 可以携带语言用于未来筛选和搜索，但 M6 不给 Note 建立独立详情页或语言版本编辑流。
- M6 不把 Dashboard 做成专业翻译管理平台。Settings 页面保留个人发布 App 的低到中信息密度，复杂编辑放进可搜索列表、Modal 和右侧小组件。

## 4. Dashboard 自身 i18n

Dashboard 自身 i18n 属于 Home Server UI 能力，不属于前台 Theme Contract。

### 4.1 资源形态

Dashboard 文案使用 JSON 资源文件维护，首批资源：

```text
Src/HomeServer/Bocchi.HomeServer/Localization/Dashboard/
  zh-CN.json
  en-US.json
```

资源 key 使用稳定语义命名，按页面或组件分组：

```json
{
  "nav.overview": "Overview",
  "nav.content": "Content",
  "settings.localization.title": "Localization",
  "settings.localization.primaryLanguage": "Primary language"
}
```

### 4.2 选择与持久化

- Dashboard UI language 的入口放在 Dashboard 顶栏账号菜单或 `Settings / Localization` 中。
- 控件使用紧凑下拉或可搜索选择框，不使用 `中文 / English` Toggle。
- 选择结果优先作为当前 Admin 的 UI 偏好保存；如果实现阶段还没有用户偏好表，可以先使用 ASP.NET Core culture cookie。
- Dashboard UI language 不写入 `theme-context.json`，也不影响前台站点输出。

## 5. 站点语言设置

M6 新增 `Settings / Localization`。`Settings / Theme` 继续只负责当前 Theme、Theme manifest、Theme 私有配置和 Theme 私有 i18n key 覆盖。

### 5.1 设置内容

`Settings / Localization` 包含：

- Dashboard UI language。
- Site primary language。
- Site enabled languages。
- Custom languages。
- Common i18n text overrides。
- Content localization URL policy。

M6 的 URL policy 固定为 `PrimaryUnprefixed`：

```text
Primary language:
  /about/
  /posts/2026/hello-bocchi/

Other enabled languages:
  /zh-TW/about/
  /zh-TW/posts/2026/hello-bocchi/
  /ja-JP/about/
```

### 5.2 Language record

后台描述语言时使用统一结构：

```json
{
  "code": "zh-CN",
  "nativeName": "简体中文",
  "englishName": "Simplified Chinese"
}
```

内置语言列表写在代码中，覆盖常见语言。用户自定义语言存入 Home Server 数据库，并在 Picklist 中与内置语言一起展示。

### 5.3 Language Picklist

Dashboard 提供可复用 Picklist 组件：

- 可选择、可搜索、可键盘操作。
- 搜索匹配 `code`、`nativeName`、`englishName`。
- 列表项同时展示当地语言名称和英文名称，例如 `简体中文 / Simplified Chinese`。
- 添加启用语言时不能重复选择已有语言。
- Site enabled languages 必须包含 Site primary language；当主要语言变更时，Dashboard 自动确保它在启用列表中。

### 5.4 自定义语言管理

用户可以添加自定义语言：

- `code` 必填。
- `nativeName` 必填。
- `englishName` 必填。
- Dashboard 提示建议使用 BCP 47 tag。
- Dashboard 不因语言代码不在内置列表中而拒绝保存。

这些自定义语言原样传递给 Theme。Theme 是否支持对应语言，由 Theme 自己决定。

## 6. 前台 i18n key 约定

前台 i18n key 是站点能力，不是 Dashboard 自身文案。

### 6.1 Common i18n key

Bocchi 提供一组通用 key 约定，Dashboard 默认内置 key 列表和默认值。Theme 可以使用这些 key，也可以不用。

初始 key 分组：

```text
menu.home
menu.posts
menu.works
menu.notes
menu.friends
menu.about
common.readMore
common.backHome
common.previous
common.next
content.translationNotice
content.viewOriginal
```

Dashboard 的覆盖列表默认为空。用户需要覆盖某个 key 时，新增一个覆盖项：

- 通过可搜索下拉选择内置 key，或输入自定义 key。
- 展开后显示 Site enabled languages 对应的文本输入。
- 未填写的语言回退到 Theme 或 Bocchi 默认值。
- 只保存用户实际填写的覆盖值。

### 6.2 Theme i18n key

Theme 可以在 manifest 中声明自己的私有 i18n key：

```json
{
  "i18n": {
    "supportedLanguages": ["en-US", "zh-CN", "zh-TW", "ja-JP"],
    "keys": [
      {
        "key": "theme.defaultStatic.colophonBuiltWith",
        "title": "Colophon built-with text",
        "description": "Footer text used by Bocchi Mono.",
        "defaultValues": {
          "en-US": "Built with Bocchi.",
          "zh-CN": "由 Bocchi 构建。"
        }
      }
    ]
  }
}
```

Dashboard 在 `Settings / Theme` 中展示当前 Theme 声明的私有 key，并允许 Admin 覆盖。Theme 私有覆盖按 `themeId + key + language` 保存，切换 Theme 后不混用。

### 6.3 覆盖解析顺序

前台 Theme 解析文案时按以下顺序取值：

1. 当前 Theme 的用户覆盖值。
2. Common key 的用户覆盖值。
3. 当前 Theme manifest 中的默认值。
4. Bocchi 通用 key 默认值。
5. key 本身。

语言回退顺序：

1. 当前页面语言。
2. Site primary language。
3. Theme manifest 的默认语言。
4. key 本身。

### 6.4 显示值引用

需要在单一字符串字段中表达“这个展示值应进入前台 i18n 解析”时，使用显示值引用：

```text
i18n://common@menu.home
i18n://theme@theme.defaultStatic.pageTemplate.normal
```

规则：

- `i18n://` 是通用前缀，表示该字段是显示值引用而不是最终展示文本。
- `common@` 指向 Bocchi 通用前台文案 key。
- `theme@` 指向当前 Theme manifest 提供的私有文案 key。
- 本轮只把该约定用于 Dashboard 展示、Theme manifest 的 Page template / special page `displayName` 和 Menu label，不做全字段全局改造。
- 解析不到时，Dashboard 保留原始 key 或 name 作为 fallback，不丢弃用户已保存的值。

## 7. Theme Contract 扩展

M6 在 `cache/theme-input/theme-context.json` 中增加 `localization` 节点：

```json
{
  "data": {
    "site": {
      "language": "en-US"
    },
    "localization": {
      "primaryLanguage": "en-US",
      "enabledLanguages": [
        {
          "code": "en-US",
          "nativeName": "English",
          "englishName": "English"
        },
        {
          "code": "zh-TW",
          "nativeName": "繁體中文",
          "englishName": "Traditional Chinese"
        }
      ],
      "urlPolicy": "PrimaryUnprefixed",
      "text": {
        "menu.home": {
          "en-US": "Home",
          "zh-TW": "首頁"
        }
      }
    }
  }
}
```

内容输入模型为 Post / Page / Work 增加语言字段和 localization 字段：

```json
{
  "id": "posts/2026/hello-bocchi@zh-TW",
  "slug": "hello-bocchi",
  "language": "zh-TW",
  "siteRelativeUrl": "/zh-TW/posts/2026/hello-bocchi/",
  "canonicalUrl": "https://example.com/zh-TW/posts/2026/hello-bocchi/",
  "localization": {
    "groupId": "posts/2026/hello-bocchi",
    "isTranslation": true,
    "sourceLanguage": "en-US",
    "sourceContentId": "posts/2026/hello-bocchi@en-US",
    "alternates": [
      {
        "language": "en-US",
        "hreflang": "en-US",
        "url": "/posts/2026/hello-bocchi/",
        "href": "https://example.com/posts/2026/hello-bocchi/",
        "title": "Hello Bocchi"
      },
      {
        "language": "zh-TW",
        "hreflang": "zh-TW",
        "url": "/zh-TW/posts/2026/hello-bocchi/",
        "href": "https://example.com/zh-TW/posts/2026/hello-bocchi/",
        "title": "Hello Bocchi"
      }
    ]
  }
}
```

Theme 可以只使用 `siteRelativeUrl` 和 `alternates`，不需要自己推导 URL。

M6 同时扩展 Theme manifest 的 Page contract：

- `pageTemplates[]` 使用 `{ name, displayName }`，用于声明可编辑 Page 可选择的模板。`normal` 强制存在；Theme 未声明时 Home Server 合成 fallback。
- `specialPages[]` 使用 `{ name, displayName, route }`，用于声明 Theme 自己提供、无需用户编辑 Markdown 的特殊页面。`route` 必须是站点根相对路径。
- Page frontmatter 只保存 `template` name，不保存 Theme id 或关联记录；切换 Theme 后，如果当前 template 不可用，Dashboard 仍显示原 name 并给出警告。
- Menu 可以指向 Theme special page；Home Server 只保存 target，Generator 只在 target 可解析时把它写入公开 Theme input。

## 8. 内容本地化模型

### 8.1 内容文件约定

Post / Page / Work 保留 M2 的目录形式，并强制同一个 localization group 放在同一个内容目录下。语言变体使用 `index.{language}.md` 文件名表达，避免在年份目录或 `pages/` 根目录下把同一内容的不同语言散成多个 sibling 目录。

```text
workspace/
  posts/
    2026/
      hello-bocchi/
        index.md
        index.zh-TW.md
        index.ja-JP.md
        assets/
          cover.jpg
  pages/
    about/
      index.md
      index.zh-CN.md
```

`index.md` 是默认变体。没有显式 `language` 时，它使用 Site primary language。

语言变体 frontmatter 示例：

```yaml
---
title: "Hello Bocchi"
slug: "hello-bocchi"
language: "zh-TW"
localization:
  group: "posts/2026/hello-bocchi"
  translationOf:
    language: "en-US"
---
```

规则：

- `language` 可省略，省略时等于 Site primary language。
- `localization.group` 可省略，省略时由内容类型、年份和目录 slug 推导。
- `localization.translationOf` 存在时，该 variant 是 Translation variant。
- 同一个 group 内，同一种 language 只能有一个 variant。
- Route 使用 group 的 canonical slug，不使用翻译标题推导路径。
- Variant 之间共享同一内容目录下的 `assets/`，避免翻译版本复制媒体。
- Dashboard 新增语言版本时，只在当前 group 目录下创建 `index.{language}.md`，不在年份目录下创建新的并列内容目录。

### 8.2 数据层语义

内容扫描后形成：

```text
ContentLocalizationGroup
- groupId
- contentType
- canonicalSlug
- canonicalRouteBase

ContentVariant
- contentId
- groupId
- language
- title
- sourceRelativePath
- siteRelativeUrl
- isTranslation
- sourceLanguage?
- sourceContentId?
```

`contentId` 建议使用 `groupId@language`。例如：

```text
posts/2026/hello-bocchi@en-US
posts/2026/hello-bocchi@zh-TW
pages/about@ja-JP
```

现有 `ContentItems` 的唯一约束需要从 `Kind + ContentId` 过渡到能够表达 language variant 的形态；旧的 slug 唯一校验也需要改为 `groupId + language` 唯一。

### 8.3 原生版本与翻译版本

- Native variant 是同一 group 下平等的语言版本。
- Translation variant 记录 `sourceLanguage` 和 `sourceContentId`。
- Theme 可以基于 `isTranslation`、`sourceLanguage` 和 `alternates` 显示“当前页面由 English 翻译，查看原文”。
- Translation variant 仍然是独立 Markdown 文件，用户可以自由编辑。

### 8.4 删除与提升

Dashboard 删除内容时遵循：

- 删除普通 variant 时，只删除当前语言文件。
- 删除某个 source variant 时，如果存在 Translation variant，必须显示确认提示。
- 删除 group 中最后一个 Native variant 时，Dashboard 提示这会删除整个 localization group 及其 Translation variants。
- Dashboard 提供“提升为原生版本”动作，把某个 Translation variant 转为 Native variant。

文件删除必须由用户确认；Dashboard 不静默级联删除 Markdown 文件。

## 9. 编辑体验

默认编辑体验保持单语言：

- 新建文章、页面或作品时不显示语言步骤。
- 编辑器右侧可以显示轻量 `Language & versions` 小组件，但不强制用户交互。
- 当前内容未显式语言时，小组件显示它使用 Site primary language。

### 9.1 添加语言版本

入口按钮：`添加语言版本`。

点击后，如果当前内容有未保存修改：

- 提醒用户先保存。
- 告知当前内容会成为这个 localization group 的基础版本。

Modal 字段：

- Language：从 Site enabled languages 中选择，不能选择当前 group 已存在的语言。
- `复制当前内容作为初始内容`：默认开启。
- `这是翻译版本`：默认关闭。
- `翻译来源`：仅在勾选翻译版本时出现，选择同 group 下已有 variant，展示语言和标题。

确认后：

- Dashboard 创建新的 `index.{language}.md`。
- 写入 `language` 和必要的 `localization` frontmatter。
- 跳转到新 variant 的独立编辑界面。

### 9.2 右侧小组件

`Language & versions` 小组件显示：

- 当前语言。
- 当前 variant 是 Native 还是 Translation。
- 同 group 已有语言版本。
- 添加语言版本按钮。
- 如果是 Translation variant，显示来源语言和“查看原文 / 打开来源”入口。

该小组件不使用宽表格，移动端进入折叠区域或独立设置抽屉。

## 10. URL、SEO 与构建输出

M6 固定 `PrimaryUnprefixed` 输出策略。

示例：Site primary language 为 `en-US`，启用 `zh-TW` 和 `ja-JP`。

```text
en-US page:
  /about/
  /posts/2026/hello-bocchi/

zh-TW page:
  /zh-TW/about/
  /zh-TW/posts/2026/hello-bocchi/

ja-JP page:
  /ja-JP/about/
  /ja-JP/posts/2026/hello-bocchi/
```

每个语言页面输出：

```html
<html lang="zh-TW">
<link rel="canonical" href="https://example.com/zh-TW/about/">
<link rel="alternate" hreflang="en-US" href="https://example.com/about/">
<link rel="alternate" hreflang="zh-TW" href="https://example.com/zh-TW/about/">
<link rel="alternate" hreflang="ja-JP" href="https://example.com/ja-JP/about/">
```

规则：

- canonical 指向当前语言页面自身。
- alternate 只列出当前 group 实际存在的语言版本。
- Theme input 在 Post / Page / Work 上输出 `canonicalUrl`；`localization.alternates[]` 同时提供站点根相对 `url` 和 SEO 用的 `hreflang` / 绝对 `href`。
- 默认 Theme 提供语言切换入口，但不根据浏览器语言强制跳转。
- Preview Route Map 需要识别带语言前缀的 route，并跳转到对应 variant 编辑页。
- Sitemap 包含所有已发布语言版本，并使用 XML Sitemap 的 `xhtml:link rel="alternate"` 表达同组语言关系。
- Atom / RSS 在 M7 Feeds/Search/Publish 阶段处理语言过滤和多语言 feed 策略。

## 11. 默认 Theme 示范

`default-static` / `Bocchi Mono` 在 M6 提供基础本地化示范：

- manifest 声明支持 `en-US`、`zh-CN`、`zh-TW`、`ja-JP`。
- 使用 `theme-context.json` 中的 `localization` 和 content `alternates`。
- 在有多个语言版本时显示语言切换。
- 在 Translation variant 上显示来源提示。
- 输出 `<html lang>`、canonical、`hreflang`。
- 不展示语言图标或国旗。
- 对未知自定义语言，优先显示 Dashboard 传入的 `nativeName` / `englishName` / `code`。

默认 Theme 的视觉仍遵循 M5 `Bocchi Mono` 方向，不混入 M4 Dashboard 的粉蓝 App 风格。

## 12. Translation Provider 后续增强入口

M6 以手动编辑作为基础实现。2026-05-26 收束决策是：不在 M6 创建空的 Translation Provider 接口、空 UI 或临时入口；Provider 只保留为后续增强设计入口。

原因：

- 当前 M6 主线已经支持添加语言版本、标记 Translation variant、记录来源、公开多语言 URL / SEO / Sitemap / Preview route，并由默认 Theme 输出语言切换和翻译提示，用户价值已经闭环。
- 浏览器本地 Translator API 已有真实候选，但仍不适合作为 M6 必做 provider：MDN 仍把相关 API 标记为 Limited availability / Experimental，浏览器实现、语言对可用性、模型下载状态、权限和用户提示都需要单独产品设计。
- 第三方翻译 API 尚没有 provider 选择、API Key 配置模型、费用/限额/隐私策略和错误恢复需求。
- LLM API 尚没有 provider/settings 边界、模型选择、prompt 约束、密钥保存和内容隐私策略。
- 过早抽象会把尚未验证的 provider 差异冻结进代码，后续真实 provider 反而需要重构。

后续如果重新进入 Translation Provider，应先满足至少一个真实 provider 的接入条件：

```text
TranslationProvider
- providerId
- displayName
- executionContext: browser | server
- availability
- supportedLanguagePairs?
- translate(request)
```

Provider 展示位置建议：

- 不放在“添加语言版本”Modal 的主路径里。
- 放在新 variant 编辑界面的右侧小组件中。
- 只有当前环境可用时才显示对应 Provider。

演进顺序：

1. 手动编辑：用户自己编写翻译内容。
2. 浏览器本地翻译：检测到浏览器能力时显示，不要求用户配置。
3. 第三方翻译 API：需要用户配置 API Key。
4. LLM API：需要用户配置 API Key 和模型参数。

这不是放弃翻译能力，而是避免在没有真实 provider 的情况下提前制造空抽象。M6 完成标准只要求手动编辑闭环和 Translation variant 的元数据/输出闭合。
