# M5 Default Static Theme

本文件是 M5 阶段的“作战图”。它把 `Docs/Milestones.md` 中 M5 的“Default Static Theme”拆成可实现、可验收、可继续讨论视觉的具体任务，并把默认前台 Theme 的架构边界、默认视觉方向、Theme 配置、静态路由、构建链路和验证口径固定下来。

> 上下文恢复入口：阅读顺序为 `Docs/Architecture.md` → `Docs/Milestones.md` → `Docs/Milestones/M3/M3.md` → `Docs/Milestones/M4/M4.md` → 本文件。
>
> M5 以后，Bocchi 第一次拥有真正可用的前台站点 Theme。Dashboard 的柔和 App 风格仍以 `Docs/Milestones/M4/UI-Design.md` 为准；本文件只约束对外前台页面与 Theme Contract。

## 1. 目标

让 Bocchi 从“能生成 Theme 输入数据”进入“能生成一套真实可部署个人主页”的阶段：

1. 建立仓库内置默认 Theme，Theme id 固定为 `default-static`，显示名称为 `Bocchi Mono`。
2. 使用内置 Fluid/Liquid 风格模板 renderer 输出 HTML / CSS / 少量 JS，不把 Node.js、SvelteKit 或其他前端构建链作为默认依赖。
3. 消费 M3 已产出的 `cache/theme-input/*.json`，渲染首页、文章、页面、作品、短文和友链。
4. 固化一套默认前台视觉：克制现代、排版优先、黑白纸墨为主、少量焦橙 accent、网格清晰、响应式友好。
5. 补齐 Theme 运行链路中 M5 必需的缺口：默认 Theme 物化、`theme-context.json`、Theme 输出收集、manifest 对账和构建错误呈现。
6. 保证 Home Server `/` Preview Host 可以预览默认 Theme，Preview Toolbar 注入不需要 Theme 自己实现。

## 2. 假设与边界

### 2.1 当前假设

- M5 不改变 Home Server / Generator / Theme Contract 的大方向：Home Server 只生成标准输入、执行 Theme 构建、校验输出，不理解第三方 Theme 内部技术栈。
- 默认 Theme 的渲染器随 Home Server 分发；DataRoot 保留 `<data>/themes/default-static/` 下的 `theme.json`、`config-schema.json`、Fluid 模板和可覆盖静态资源，作为可见 Theme 实例。
- 默认 Theme 可以被复制到 DataRoot 运行数据；`<data>/workspace/` 仍不包含 Theme 实现、Theme 配置或构建产物。
- M5 只支持单语言内容。语言切换与多语言路由需要新的内容模型，不在默认 Theme 里假装实现。
- M5 继续把 RSS / Atom、Sitemap、robots 视为 Generator 产物；Theme 只链接它们，不重新生成。
- SvelteKit 仍然是未来前端 Theme 的一等公民，适合后续动态短文、Cloud JSON 拉取和复杂交互；它不再是 M5 默认 Theme 的必要依赖。

### 2.2 视觉参考边界

默认 Theme 可以吸收 mono 原型的这些方向：

- editorial / personal portfolio 的内容气质。
- 8px grid、强分隔线、克制留白、排版做主角。
- ink / paper 中性色为主，焦橙只做少量链接、hover 和 active 状态。
- 文章列表采用可扫读的行式列表；作品页才使用少量边线卡片。
- 默认无复杂动效，只有 hover、focus 和很轻的进入反馈。

默认 Theme 不照搬这些内容：

- 不复用静态原型代码、页面文案、角色图形或外部资源路径。
- 不把原型里的账号页、后台页、评论区、点赞、收藏、搜索按钮当作 M5 功能。
- 不使用负 `letter-spacing`，也不使用随 viewport 连续缩放的字体尺寸；字号用语义 token 和断点调整。
- 不添加吉祥物、角色、可识别动漫元素或装饰性 emoji。Bocchi 前台默认 Theme 应该靠内容、排版和用户媒体建立气质。

## 3. 关键结论

### 3.1 默认 Theme 名称

- Theme id：`default-static`
- 显示名称：`Bocchi Mono`
- 版本：M5 初始为 `0.1.0`
- Contract：`1.0`

`default-static` 表示默认实现的运行方式，`Bocchi Mono` 表示默认视觉名称。后续即使新增 `default-svelte` 或其他实验 Theme，也不应改变这个默认 id 的语义。

### 3.2 架构优先级

M5 不能只做一组静态页面模板。真正的完成标准是：Generator 能把默认 Theme 的输出纳入同一份 `.bocchi-manifest.json`，Home Server Preview 能打开生成后的页面，Dashboard 里选择默认 Theme 后能实际影响构建。

因此 M5 的前置工程任务是：

1. **默认 Theme 物化**：随 Home Server 分发的默认 renderer 是源；工作区 `<data>/themes/default-static/` 是运行实例，至少包含 `theme.json`、`config-schema.json` 和可覆盖的静态资源。初始化或明确安装动作只在目标不存在时复制，不覆盖用户修改。
2. **Theme Context 输入**：生成 `cache/theme-input/theme-context.json`，并把 `state/theme-config/{themeId}.json` 中的用户配置与 `config-schema.json` 默认值合并到 `theme.config`；Home Server 的 `ThemeConfigurationRecord` 作为 UI 投影，需要在保存时同步原始配置文件。
3. **Theme 输出收集**：Theme 先输出到 `theme.json.outputDir` 指定的 Theme 本地目录，再由 Generator 新增的输出收集阶段写入 `output/public/` 并登记为 `ArtifactKind.ThemeOutput`。
4. **manifest 对账**：`WriteManifestStage` 必须包含 Theme HTML / CSS / JS / assets，否则 M7 发布无法可靠对账。

这比让任何 Theme 直接写入 `output/public/` 更清晰。直接写入会绕过 `BuildArtifact`，随后被 `ValidateOutputStage` 当作未登记 orphan 文件清理，和现有 artifact 模型冲突。

### 3.3 默认 renderer 技术选择

结论：M5 默认 Theme 使用 **随 Home Server 分发的 Fluid/Liquid 风格模板 renderer**。Fluid 是默认 Theme 作者和新手改主题的第一条路，但不是 Theme Contract 的唯一实现方式。

取舍：

- **Fluid / Liquid 类模板**：轻量、可控、容易复制到工作区后由用户修改；适合给社区一个默认答案，避免一开始就碎成多个构建生态。代价是弱类型和 helper/filter 维护成本，因此它只作为默认模板路线，不进入 Theme Contract 的核心协议。
- **Blazor / Razor Components 静态输出**：组件语法熟悉，且 Home Server 已经是完整 ASP.NET Core / Blazor Server，边际依赖成本不高；但脱离请求管线做批量静态渲染仍会引入宿主、CSS isolation、路由和生命周期复杂度。它适合未来作为 `dotnet` runner 的高级 Theme，不作为 M5 默认新手路线。
- **RazorLight / runtime Razor**：模板体验好，但 runtime compilation 会把 Roslyn 编译语义带进 Theme 编辑和部署。它不适合作为默认可编辑 Theme 的第一答案。
- **手写 typed C# renderer**：最直接、强类型，但不利于社区学习和修改 Theme；可以作为 Fluid renderer 内部 helper 的实现方式，而不是用户面对的主题形式。
- **独立 AOT renderer**：隔离性好，但 Home Server 本身已经按完整 server/JIT 路线发布，M5 阶段不需要为默认 Theme 额外承担多 RID 分发和独立进程管理。

M5 推荐形态：

- 仓库 `Themes/default-static/` 是默认 Theme 的 canonical source，目录形态与第三方 Theme 一致。
- `Src/Themes/Bocchi.Theme.DefaultStatic/` 承载 `fluid-static` runner、embedded resource 物化和少量模板模型代码，不再把模板、CSS、JS、manifest 写成 C# raw string。
- 工作区 `themes/default-static/` 中物化 `theme.json`、`config-schema.json`、`templates/`、`assets/`，用户可以改模板和 CSS。
- renderer 通过清晰接口读取 `cache/theme-input/theme-context.json` 与内容输入，写入 Theme 本地 `outputDir`，不访问数据库、不依赖 Home Server 页面组件。
- 由 `fluid-static` runner 在 Home Server 进程内调用这个可信 renderer；第三方纯模板 Theme 也可以使用同一 runner，后续如需隔离再拆成独立进程或 AOT executable。
- HTML 输出默认由 Fluid auto-escape；确实要输出正文 HTML 时必须经过明确的 `html` filter / helper，避免模板作者无意绕过安全边界。

### 3.4 Runner 边界

Runner 是 Theme Contract 的执行层，不是某一种技术栈。M5 本地 runner 至少支持：

- `fluid-static`：调用 Bocchi 随包分发的 Fluid 静态站点 renderer，服务默认 `default-static` Theme，也给第三方纯模板 Theme 使用。
- `process`：按 `theme.json` 声明执行任意本地命令，给 SvelteKit / Astro / Hugo / 自定义 binary / Blazor static renderer 留入口。

所有 runner 都遵守同一组约束：

- 读取 `BOCCHI_INPUT_DIR`。
- 写入 `BOCCHI_OUTPUT_DIR`，且该目录是 Theme 本地输出目录，不是最终 `output/public/`。
- stdout / stderr 回传为 Build 日志。
- 失败、超时、取消统一转换成 Theme runner 错误。
- Generator 在 runner 结束后统一收集输出、登记 artifact、写 manifest。

### 3.5 功能不伪装

M5 不做还没有数据支持的交互：

- 不显示搜索入口；搜索索引属于 M7。
- 不显示评论、点赞、收藏、阅读量；这些需要 Cloud Server 或第三方动态能力。
- 不显示语言切换；当前内容模型没有多语言字段。
- 不显示不可用按钮、占位设置或“以后会做”的控件。

## 4. 交付范围

### 4.1 在范围内

- 默认 Theme renderer：
  - `theme.json`
  - `config-schema.json`
  - Fluid 模板 renderer 项目
  - `templates/`
  - `assets/`
  - Theme 输入读取层
  - 全局 layout、页面片段、基础样式和小型前端脚本
- 默认页面：
  - 首页 `/`
  - 文章列表 `/posts/`
  - 文章详情 `/posts/{year}/{slug}/`
  - 独立页面 `/{slug}/`
  - 作品列表 `/works/`
  - 作品详情 `/works/{year}/{slug}/`
  - 短文总览 `/notes/`
  - 短文年份页 `/notes/{year}/`
  - 友链 `/friends/`
  - `404.html`
- Generator / Workspace 集成：
  - 默认 Theme 物化或安装路径
  - `theme-context.json` 写入
  - Theme 本地输出目录清理
  - Theme 输出收集为 `ArtifactKind.ThemeOutput`
  - manifest 和 validate 与 Theme 输出一致
- Home Server 集成：
  - Setup / Workspace 初始化后默认主题可用
  - Settings 中默认 Theme 配置保存后参与构建
  - Build / Preview 页面可以跑通默认 Theme
- 验证：
  - .NET 单元/集成测试覆盖新增 Generator 行为
  - Theme 构建 smoke 覆盖空内容、完整样例内容、草稿预览
  - 关键视口截图或手工验收记录

### 4.2 非目标

- 不做多套视觉主题。
- 不做 Theme marketplace。
- 不做 Theme 热更新、远程下载或不可信沙箱。
- 不做搜索索引生成和搜索 UI。
- 不做评论、点赞、收藏、访问统计。
- 不做图片派生、响应式图片生成或 OG image 生成。
- 不做多语言内容模型。
- 不做细粒度增量渲染；M5 仍然允许整站 Theme build。
- 不做 GitHub Actions Remote Runner；它归入 M7 发布管线规划。

## 5. Theme Contract 与构建链路

### 5.1 目录结构

仓库内置 Theme 源：

```text
  config-schema.json
  templates/

  assets/

```

仓库内置 runner / 物化代码：

```text

  DefaultStaticThemeDefinition.cs
  DefaultStaticThemeResources.cs
  DefaultStaticTemplateRenderer.cs
  DefaultStaticFluidRenderer.cs
```

DataRoot 运行实例：

```text
<data>/

      theme.json
```

### 5.2 theme.json

`theme.json` 初始建议：

```json
{
  "id": "default-static",
  "name": "Bocchi Mono",
  "version": "0.1.0",
  "contractVersion": "1.0",
  "inputDir": "../../cache/theme-input",
  "outputDir": "build",
  "runner": {
    "kind": "fluid-static",
    "entry": "fluid"
  },
  "features": {
    "posts": true,
    "pages": true,
    "works": true,
    "notes": true,
    "friends": true,
    "photos": false,
    "search": false
  }
}
```

`search` 初始为 false。即使 `SiteSettings.EnableSearch` 已有字段，默认 Theme 在 M5 不显示搜索入口，避免用户看到不可用功能。

兼容说明：现有代码中的 `build.command` 是 M3 的 process runner 形态。M5 可以用一次小的 Theme Manifest 迁移把它收束为 `runner.kind = process`；如果实现时为了降低改动量，也可以先让 `build.command` 作为 `process` runner 的兼容别名，但新文档与新 Theme 都以 `runner` 为准。

### 5.3 Theme 输入读取

Theme 读取优先级：

1. `BOCCHI_INPUT_DIR` 环境变量。
2. Theme 根目录下 `theme.json.inputDir`。
3. 开发 fallback：Theme 根相对路径 `../../cache/theme-input`。

读取规则：

- 每个输入文件先解开 `InputEnvelope<T>`。
- `contractVersion` 必须等于 `1.0`，否则构建失败并给出清晰错误。
- `data` 做轻量结构断言，不重复 Generator 已经做过的内容校验。
- 文章、作品、短文的排序在 Theme 侧完成，用 `publishedAt ?? updatedAt` 降序；缺日期的内容排在后面。
- 正文默认使用 `html`，不在 Theme 中二次解析 Markdown。代码高亮、数学公式和图片灯箱以后可以作为 Theme 内部增强，但 M5 不先扩协议。

### 5.4 theme-context.json

M5 正式引入 `cache/theme-input/theme-context.json`。它是 Theme 渲染时的全局上下文，不是扁平的“全局变量表”，而是结构化的 Theme Context：

```json
{
  "$schema": "https://bocchi.local/schema/v1/theme-context.json",
  "contractVersion": "1.0",
  "generatedAt": "2026-05-15T00:00:00Z",
  "data": {
    "bocchi": {
      "version": "0.1.0"
    },
    "build": {
      "generatedAt": "2026-05-15T00:00:00Z",
      "environment": "production",
      "includeDrafts": false
    },
    "site": {
      "title": "My Site",
      "version": null,
      "description": "",
      "language": "zh-CN",
      "timeZone": "Asia/Shanghai",
      "baseUrl": "https://example.com/"
    },
    "author": {
      "name": "Anonymous",
      "displayName": "Anonymous",
      "timeZone": "Asia/Shanghai",
      "email": null,
      "bio": null,
      "links": []
    },
    "features": {
      "rss": true,
      "sitemap": true,
      "search": false
    },
    "theme": {
      "id": "default-static",
      "name": "Bocchi Mono",
      "version": "0.1.0",
      "config": {
        "visual": {
          "accentColor": "#E85D3A",
          "paperTone": "warm"
        },
        "home": {
          "featuredPosts": 5,
          "featuredWorks": 4,
          "recentNotes": 3,
          "showFriends": true
        }
      }
    }
  }
}
```

上下文原则：

- `theme-context.json` 聚合站点事实、作者信息、构建信息、功能开关和当前 Theme 的有效配置。
- `theme.config` 是 `config-schema.json` 默认值与 `state/theme-config/{themeId}.json` 用户配置合并后的结果。
- 原始用户配置继续保存在 `state/theme-config/{themeId}.json`；Theme 渲染优先读取 `theme-context.json`。
- 配置控制“展示策略”，不复制 `site.yaml` 已经拥有的站点事实。
- 颜色只开放 accent，不开放整套 token。默认 Theme 先保持一致审美，不做主题编辑器。
- `reading.showReadingTime` 初始 false，因为当前 Contract 没有字数或阅读时间字段；开启时只能由 Theme 从纯文本估算，验收前要确认是否值得做。

### 5.5 config-schema.json

`config-schema.json` 是 Theme 告诉 Home Server Dashboard “我希望暴露哪些配置项”的声明文件。初始 schema 分三组：

- `visual`：accent color、paper tone、是否显示首页媒体区域。
- `home`：首页文章/作品/短文数量、是否显示友链。
- `reading`：更新时间、目录、阅读时间估算。

字段模型：

- 字段必须有 `key`、`type`、`title`。
- 字段可以有 `description`、`placeholder`、`default`、`required`、`helpText`。
- 条件显示只允许结构化 predicate，不允许任意 JS 表达式。
- 图片字段使用 `image` type，允许声明 `accept`、`recommendedSize` 和用途说明。

示例：

```json
{
  "schemaVersion": "1.0",
  "groups": [
    {
      "id": "home",
      "title": "首页",
      "fields": [
        {
          "key": "home.showHeroImage",
          "type": "boolean",
          "title": "显示首页图片",
          "description": "在首页首屏显示一张图片。",
          "default": false
        },
        {
          "key": "home.heroImage",
          "type": "image",
          "title": "首页图片",
          "description": "建议使用横图。",
          "placeholder": "选择图片",
          "accept": ["image/jpeg", "image/png", "image/webp"],
          "recommendedSize": { "width": 1600, "height": 900 },
          "visibleWhen": {
            "key": "home.showHeroImage",
            "equals": true
          }
        }
      ]
    }
  ]
}
```

`visibleWhen` 支持 `key` + `equals` / `notEquals` / `in`，以及 `all` / `any` 组合。M5 不支持任意表达式，避免 Dashboard 表单逻辑变成脚本执行环境。

Schema 只描述 M5 真能执行的配置；不放搜索、多语言、评论或统计开关。

### 5.6 默认 renderer 分发

M5 默认 renderer 随 Home Server 分发，不需要 install 阶段。

建议行为：

- 默认 Theme 构建不触发网络访问。
- 默认 Theme 构建不需要 Node.js、pnpm、npm、Roslyn runtime compilation 或额外可执行文件。
- 默认 Theme 使用 Fluid 模板，模板和 CSS 可复制到工作区修改。
- 第三方 Theme 的 `process` runner 可以声明 `installCommand`，但它必须是显式动作，不在每次 Full Build 自动执行。
- Full Build 发现第三方依赖缺失时给出可执行提示，而不是把底层工具长堆栈直接抛给普通用户。
- 后续如果要自动安装第三方依赖，必须用 lockfile hash 做幂等判断，并在 UI 上明确这是执行本机代码。

### 5.7 SvelteKit 的保留位置

SvelteKit 不作为 M5 默认 Theme，但保持一等支持方向：

- `process` runner 能运行 `pnpm build`，因此第三方 SvelteKit Theme 不需要特殊协议。
- 后续短文 / 推文动态化、Cloud JSON 拉取、Mastodon 生态对接，更适合在某个 SvelteKit Theme 或 Cloud 前端里落地。
- M5 的默认静态 Theme 不阻断这条路；它只是避免 Home Server 默认 Docker 镜像因为一个基础 Theme 被迫携带 Node.js。

## 6. 页面与路由设计

### 6.1 首页 `/`

首页不是营销 landing page，而是个人主页入口。首屏应立即表达站点主人、写作方向和最近内容。

结构：

1. 顶栏：wordmark、导航、前台外观按钮；不放搜索。
2. Hero：站点标题、站点描述、作者简介或一句引言。
3. Selected writing：最新或精选文章行式列表。
4. Selected work：精选作品边线卡片。
5. Recent notes：最近短文流。
6. Colophon：站点元信息、RSS / Atom、社交链接。

如果存在 slug 为 `index` 的 Page，可以作为首页正文补充，但不能替代站点标题和内容入口。

### 6.2 文章列表 `/posts/`

默认使用行式列表：

```text
2026 - 05   文章标题                             分类 / tag
2026 - 04   另一篇文章                           分类 / tag
```

桌面端按年/月、标题、分类/标签分列；移动端折成两行，保持触摸区域足够大。

### 6.3 文章详情 `/posts/{year}/{slug}/`

结构：

- 返回 Writing 的轻链接。
- 标题、摘要、发布时间、更新时间、分类、标签。
- 可选 cover，使用稳定 aspect-ratio。
- 正文宽度约 680px。
- 文章尾部显示标签、上一篇/下一篇。

不显示评论、点赞、阅读量、收藏按钮。没有数据来源的社交互动不进入 M5。

### 6.4 独立页面 `/{slug}/`

独立页面服务 About、Uses、Now 等长期页面：

- 使用比文章更简单的页面头。
- 不显示发布时间。
- 支持 `showInNavigation` 进入顶栏。
- 保留正文同样的 prose 样式。

路由冲突处理：

- `/posts/`、`/works/`、`/notes/`、`/friends/` 为保留路径。
- Page slug 不能覆盖保留路径；如果内容层没有阻止，Theme 构建阶段要报清晰错误。

### 6.5 作品列表与详情

`/works/` 使用边线卡片，但不做大投影或复杂网格：

- 标题。
- 一句话 summary。
- period / role。
- stack 文本标签。
- links。
- 可选 cover。

`/works/{year}/{slug}/` 使用作品详情模板，正文仍来自 `html`，上方强化作品元数据。

### 6.6 短文 `/notes/` 与 `/notes/{year}/`

短文总览按年份分组，年份页展示具体条目：

- 时间戳使用 mono 字体。
- 每条短文用正文 + 媒体 + tag + 分隔线，不做卡片墙。
- 短文没有单条详情页，除非后续内容模型显式需要 permalink。

### 6.7 友链 `/friends/`

友链页默认是文本优先的目录：

- name。
- description。
- url。
- tags。
- avatar 可选；没有 avatar 时不能出现破图或占位大图。

友链 status 不是 `published` 时默认不展示，除非 BuildContext `includeDrafts` 为 true。

### 6.8 404

`404.html` 使用同一套顶栏和视觉 token，但保持克制：

- 简短说明。
- 回首页和 Writing 的链接。
- 不做满屏插画或游戏化交互。

## 7. 视觉系统

### 7.1 色彩

语义 token：

| Token | Light | Dark | 用途 |
| --- | --- | --- | --- |
| `--bg` | `#FAFAF7` | `#0A0A0B` | 页面背景 |
| `--surface` | `#FFFFFF` | `#141416` | 少量表面 |
| `--surface-soft` | `#F1F1EC` | `#1A1A1C` | hover / 区块 |
| `--text` | `#101012` | `#EDEDEA` | 主文字 |
| `--text-muted` | `#52524F` | `#A8A8A4` | 次要文字 |
| `--text-faint` | `#9A9A95` | `#6A6A66` | 弱信息 |
| `--line` | `#E5E5E0` | `#26262A` | 分隔线 |
| `--accent` | `#E85D3A` | `#FF7A55` | 链接 / active / focus |
| `--accent-soft` | `#FFE9E0` | `#3A1B11` | 极少量提示底色 |

规则：

- accent 面积控制在全站很小比例；首屏不能被焦橙吞掉。
- 默认背景不能变成米色系一把梭；用户图片、cover 和内容媒体应承担大部分视觉变化。
- Dark 是低刺激阅读模式，不是终端主题。

### 7.2 字体与排版

默认字体栈：

```css
--font-display: "Instrument Serif", "Noto Serif SC", ui-serif, Georgia, serif;
--font-body: "Inter Tight", "Inter", "Noto Sans SC", ui-sans-serif, system-ui, sans-serif;
--font-mono: "JetBrains Mono", ui-monospace, SFMono-Regular, Menlo, monospace;
```

M5 实现时要提供系统字体 fallback。外部字体加载失败时页面仍必须可读。

字号原则：

- Hero / article title 可以使用 display 字体，但只在真正的大标题中使用。
- 列表、卡片、导航、footer 使用 body 或 mono。
- 不使用负 letter-spacing。
- 不用 `vw` 直接连续缩放字体；按断点切换语义字号。

### 7.3 布局

- baseline：8px。
- 顶栏高度：56px。
- 最大容器：1280px。
- 主内容：1080px。
- 正文宽度：680px。
- 卡片圆角最大 8px。
- 页面分隔主要依靠 1px line，不依赖大投影。
- 固定格式元素必须有稳定尺寸，避免 hover、长标题或图标加载导致布局跳动。

### 7.4 响应式

检查视口：

- `1440x900`
- `1024x768`
- `768x1024`
- `412x915`

移动端规则：

- 顶栏导航折叠为菜单。
- 列表行折成两行，不变成压缩表格。
- 作品卡片单列。
- 正文左右 padding 保持可读，长 URL / 长英文单词可换行。
- 前台外观按钮仍是图标按钮，但必须有 accessible label。

### 7.5 动效

- 默认无入场大动画。
- hover / focus / active 使用 120-200ms。
- 尊重 `prefers-reduced-motion`。
- 不做滚动视差、全屏过渡或复杂页面切换。

### 7.6 前端渐进增强

默认 Theme 的 JS 路线是“完整静态 HTML + 原生渐进增强”：

- 没有 JS 时，所有页面内容、导航、时间、链接仍可读可用。
- `app.js` 使用原生 ES module、`data-*` attribute 和少量 Web Components，不引入默认前端框架。
- 首批组件只做明确小能力，例如 `bocchi-time`、移动端菜单、外观切换。
- 后续交互变复杂时，第三方 Theme 可以用 SvelteKit / Vue / Astro；默认 Theme 不提前承担大型框架。

默认 Theme 必须实现双时区时间增强：

```html
<time
  datetime="2026-05-15T10:30:00+08:00"
  data-bocchi-time
  data-author-time-zone="Asia/Shanghai">
  2026-05-15 10:30 CST
</time>
```

行为：

- 静态 HTML 默认显示作者时区时间。
- 浏览器用 `Intl.DateTimeFormat().resolvedOptions().timeZone` 获取访问者时区。
- 访问者时区与作者时区一致时，保持普通显示。
- 时区不一致时，追加一个轻量时区标记；hover / click 时间或标记时，显示作者时区 + 访问者时区的双时区提示。
- 优先使用原生 Popover API；不支持时退化为 `title` 或轻量 positioned tooltip。
- 该能力约束默认 Theme 至少具备基本前端程序能力，同时不破坏纯静态输出。

有帮助的前端方向：

- Progressive enhancement：先输出完整 HTML，再增强。
- Web Components / Custom Elements：适合 `bocchi-time` 这类跨 Theme 小组件。
- Popover API：适合双时区提示、轻量菜单和说明浮层。
- CSS container queries / `:has()`：减少响应式与状态样式对 JS 的依赖。
- SvelteKit：保留给动态短文、Cloud JSON 拉取、Mastodon 生态等高级 Theme。

## 8. 与 Home Server Preview 的关系

Preview Toolbar 仍由 Home Server Preview Host 注入，默认 Theme 不包含后台返回按钮或编辑按钮。

Home Server Live Preview 是正式构建模式，不是临时绕过流程：

- `theme-context.json` 的 `build.mode` 使用 `full` / `live` 区分静态发布构建与实时预览构建。
- Live Preview 仍走同一套 Theme Contract 输入、runner 和 Theme 输出收集；区别只是 `BOCCHI_INPUT_DIR` / `BOCCHI_OUTPUT_DIR` 指向本次请求的一次性 cache 目录。
- Theme 必须只读取 `BOCCHI_INPUT_DIR`，只写 `BOCCHI_OUTPUT_DIR`。不得读取旧的 Full Build 输出，也不得直接写 `output/public/`。
- 内部链接、CSS、JS、媒体路径默认使用站点根相对路径，例如 `/assets/app.css`、`/media/...`；canonical / hreflang 使用公开 `site.baseUrl`。
- Post / Page / Work 的 canonical 内容 route 应来自 Theme Contract 输入中的 `siteRelativeUrl`。`url` 仅作为 v1 兼容别名保留，后续多语言内容版本会继续把 route/alternate 关系放进输入数据。

Theme 需要配合的点：

- 顶栏 z-index 不应高到盖住 Preview Toolbar。
- HTML 结构应保留正常 `<main>`，便于注入脚本定位。
- 不用全屏 fixed overlay 作为常规布局。
- 所有内容页都使用稳定的 canonical route，便于 Home Server 根据 Theme Contract URL 生成 Preview Edit 映射；自定义额外页面可预览，但不承诺显示 Edit 按钮。

## 11. 风险与处理

| 风险 | 处理 |
| --- | --- |
| Theme 直接写入 `output/public/` 导致 artifact/manifest 失真 | M5 改为 Theme 本地输出 + `CollectThemeOutputStage` |
| 默认 Theme 只存在 C# 内置模块中，第三方作者无正式参考 | `Themes/default-static/` 成为 canonical source，发布时嵌入资源，运行时物化 |
| Theme 设置只存在 SQLite，CLI build 读不到 | 以 `state/theme-config/{themeId}.json` 作为构建源，SQLite 作为 UI 投影 |
| 视觉原型里的不可用功能被搬进默认站点 | M5 明确不显示搜索、评论、点赞、语言切换 |
| 字体或外部 CDN 不可用 | 提供系统字体 fallback；不依赖远端脚本完成首屏 |
| 用户内容为空时页面尴尬 | 首页和列表页提供克制 empty state，提示从 Dashboard 写第一篇内容 |
| 默认 renderer 变成 Home Server 内部杂糅代码 | 单独项目 + 输入/输出接口隔离；不访问数据库和 Blazor 页面组件 |
| Fluid helper/filter 膨胀成小框架 | 只提供 URL、日期、摘要、asset、安全 HTML 等少量稳定 helper；复杂交互交给 process runner Theme |
| `config-schema.json` 条件逻辑失控 | 只支持结构化 `visibleWhen` predicate，不执行任意表达式 |
| 第三方 Theme 依赖安装失败信息太技术化 | Build 页面将常见缺依赖错误翻译成可执行提示 |
