# M4 Home Server Dashboard

本文件是 M4 阶段的“作战图”。它把 `Docs/Milestones.md` 中 M4 的“Home Server Dashboard”拆成可恢复、可验收、可反复讨论 UI 的具体任务，并把本阶段新增的正式 ASP.NET Core 用法、用户系统、Setup 流程、Dashboard 信息架构、Markdown 编辑器和前台预览模式固定下来。

> 上下文恢复入口：阅读顺序为 `Docs/Architecture.md` → `Docs/Milestones.md` → `Docs/Milestones/M1/M1.md` → `Docs/Milestones/M2/M2.md` → `Docs/Milestones/M3/M3.md` → 本文件。
>
> M4 以后，Dashboard 不再按“临时后台壳”推进，而是按正式但亲和的个人发布后台推进。若本文件与更早文档冲突，以本文件、`Docs/Milestones/M4/UI-Design.md` 和用户在 2026-05-14 对 M4 的细化说明为准。

## 1. 目标

让 Home Server 从“能触发构建的内网页面”升级为日常可用、可长期演进的正式内容管理后台：

1. 使用正式 ASP.NET Core 方式承载 Home Server：ASP.NET Core Identity、EF Core、SQLite、角色授权、外部登录、设置页和受保护路由。
2. 首次启动进入 Welcome / Setup，引导初始化数据库、创建第一个 Admin 账户，并在完成后关闭公开注册。
3. Dashboard 以 `/Admin` 为基础 URL，前台站点预览以 `/` 为基础 URL；整个 Home Server 默认需要登录，前台站点在 Home Server 中也只是受保护的实时预览。
4. Dashboard 采用正式个人 Blog / CMS 的信息架构，但视觉体验按柔和、低到中信息密度、App-like 的个人发布工具推进，支持内容列表、详情、Markdown 分栏编辑、站点设置、Theme 配置、发布/构建状态、用户管理和第三方登录设置。
5. UI 阶段必须集中安排：先确认视觉风格、布局、组件、移动端模式、Dark / Light / Auto 主题，再进入主要功能开发。
6. 前台预览模式可以叠加浮动工具栏，用于表达“当前是 Preview”、返回 Dashboard，并在文章 / 页面 / 作品等详情页提供编辑入口。

## 2. 关键结论

### 2.1 M4 的技术方向

- Home Server 继续使用 **Blazor Web App + InteractiveServer**，这是 M1 已确认的后台 UI 方向。
- M4 引入 **ASP.NET Core Identity** 作为用户系统，不自研账户、密码、登录 cookie 或角色表。
- M4 引入 **EF Core + SQLite** 作为 Home Server 的正式数据访问方式。早期 `Microsoft.Data.Sqlite` + 手写 `SchemaMigrator` 是 M2/M3 期间为内容扫描和构建状态打通闭环的实现形态，M4 起应收束到 EF Core migration 与 `DbContext` 模型，不继续扩大手写 SQL 的边界。
- 内容事实仍然是 Markdown / YAML / 原始媒体文件。EF Core 不存储文章、页面、作品的正文，只存储 Identity、设置、索引、状态、构建记录、外部登录配置和 UI 偏好。
- Dashboard 视觉不做营销站，不做大 Hero，不做装饰型卡片堆叠；它应当像一个柔和的个人发布 App：清晰、低到中信息密度、普通用户能理解，同时保留长期写作、改稿、预览和发布所需的稳定信息架构。

### 2.2 预览工具栏可行性

结论：可以做，而且应该由 Home Server 的 Preview Host 统一注入，而不是要求每个 Theme 自己实现工具栏。

可行方案：

1. `/` 下的前台站点只在 Home Server 内作为受保护预览存在，不作为公开站点入口。
2. HTML 响应由 Preview Host 端点返回，而不是直接裸跑 `UseStaticFiles` 公开整个输出目录。
3. 对 HTML 页面，Preview Host 在响应末尾注入一段很小的 toolbar script / style，或把页面包在一个 preview shell 中。
4. 对 CSS、JS、图片、字体等非 HTML 资源，Preview Host 通过受授权的文件端点返回，不注入工具栏。
5. 工具栏读取当前 route 到 content id 的映射；能定位到文章 / 页面 / 作品时显示“编辑”按钮，不能定位时只显示“返回 Dashboard”和预览状态。

这条路与 ASP.NET Core 的授权模型一致：静态资源如果要受授权保护，应放在 web root 之外，并在授权中间件之后通过受保护端点或文件结果返回。它也避免把工具栏变成 Theme Contract 的强制内容。

### 2.3 外部登录配置可行性

结论：GitHub 和通用 OpenID Connect Provider 都可以接入 ASP.NET Core Identity，但 M4 必须把“可配置 Provider”设计为后台设置，而不是写死 Logto 或写死配置文件。

- GitHub 使用 OAuth provider；如果没有内置 GitHub provider，则使用社区维护的 ASP.NET Core OAuth provider 包，并把它视为 GitHub 这一单一 provider 的 adapter。
- OpenID Connect 使用 ASP.NET Core 的 OIDC middleware，配置字段保持通用：Authority / ClientId / ClientSecret / CallbackPath / Scopes / ResponseType / Claim Mapping。Logto 只是一个示例配置，不进入命名或数据模型。
- 登录页只显示已启用且配置完整的 provider 按钮。
- Provider 的 client secret 属于敏感数据：允许在 Dashboard 输入，但落库前必须用 ASP.NET Core Data Protection 保护；同时支持环境变量覆盖，便于用户不想把 secret 放进 SQLite。

## 3. 范围与非目标

### 3.1 在范围内

- **UI 风格确认阶段**：视觉关键词、低到中信息密度、App-like 内容列表、布局草图、设计 token、组件状态、移动端方案、Dark / Light / Auto、编辑器体验和预览工具栏样式。
- **Identity 与 EF Core 基础**：`BocchiDbContext`、Identity user / role、SQLite provider、EF Core migrations、Data Protection key 存储策略。
- **Welcome / Setup**：首次启动检测、数据库初始化、第一个 Admin 创建、注册关闭、初始化完成后进入 `/Admin`。
- **权限模型**：多用户底层能力、单站点、非多租户、唯一业务角色 `Admin`；非 Admin 用户没有 Dashboard 权限。
- **登录方式**：本地账号密码、GitHub、通用 OpenID Connect Provider。外部登录在设置页启用后才显示按钮。
- **Dashboard Shell**：`/Admin` 路由、顶栏、侧边栏、二级菜单、移动端二级分类下拉、Dashboard 外观 / dark mode 切换、全局状态与错误呈现。
- **内容管理**：文章、页面、作品、短文、友链、站点设置的列表 / 筛选 / 详情入口；Markdown 分栏编辑器；frontmatter 表单；保存写回内容文件。
- **构建与日志**：消费 M3 Pipeline，展示阶段进度、日志、manifest、artifact 树、zip 下载和失败定位。
- **设置面板**：基本信息、数据库、第三方登录、Theme 配置、用户管理、工作区状态。
- **预览模式**：`/` 作为受保护前台预览；浮动工具栏；Dashboard 跳转；详情页编辑按钮。
- **测试与验证**：Identity / Setup / 授权 / 外部登录配置 / Dashboard 页面 / 编辑保存 / 预览工具栏 / 移动端布局的自动化或手工 smoke。

### 3.2 非目标

- 不做多租户。Bocchi 始终只有一个站点、一个前台网站、一个内容工作区。
- 不做复杂多角色权限系统。M4 只做 `Admin` 一个业务角色。
- 不做 WYSIWYG 编辑器。正文编辑以 Markdown 为主，采用编辑区 + 预览区的分栏体验。
- 不把前台站点变成公网动态站点。Home Server 内的 `/` 是登录后的预览模式；公开静态站点仍由发布产物承担。
- 不在 M4 完成默认 SvelteKit Theme 的完整视觉。M4 负责 Preview Host 与工具栏；真正默认 Theme 的页面视觉归 M5。
- 不提前实现 Cloud Server、评论、公开 API、访问统计等动态能力。

## 4. 关键设计

### 4.1 URL 与路由边界

| URL | 含义 | 鉴权 |
| --- | --- | --- |
| `/Setup` | 首次启动 Welcome / Setup 流程 | 仅未初始化时允许匿名访问；初始化完成后永久不可用 |
| `/Account/Login` | 登录页 | 允许匿名访问 |
| `/Account/*` | Identity 登录、登出、外部登录回调、账号相关端点 | 按 Identity 默认安全策略处理 |
| `/Admin` | Dashboard 首页 | 需要登录且属于 `Admin` |
| `/Admin/Content/*` | 内容管理 | 需要 `Admin` |
| `/Admin/Settings/*` | 设置面板 | 需要 `Admin` |
| `/Admin/Users/*` | 用户管理 | 需要 `Admin` |
| `/Admin/Publish/*` | 发布、构建与日志 | 需要 `Admin` |
| `/` | 前台站点实时预览 | 需要登录；详情编辑入口需要 `Admin` |

授权策略：

- 默认 fallback policy：除明确标记匿名的 Setup / Login / 外部登录回调 / 登录页静态资产外，所有请求都需要认证。
- Dashboard policy：`RequireRole("Admin")`。
- 非 Admin 登录后只能看到无权限页或账号退出入口，不允许访问内容、设置、构建或预览编辑操作。
- 预览静态资源不得放在公开 `wwwroot` 下裸访问；应由 Preview Host 受控返回。

### 4.2 数据访问与 SQLite

M4 的目标是建立一个正式的 Home Server 应用数据库，而不是继续在多个地方分散写 SQL。

建议结构：

```text
Src/HomeServer/Bocchi.HomeServer/
  Data/
    BocchiDbContext.cs
    BocchiUser.cs
    BocchiRoleNames.cs
    SetupState.cs
    DashboardSettings.cs
    ExternalLoginProviderSettings.cs
    ThemeConfigurationRecord.cs
    BuildRunRecord.cs
    BuildStageLogRecord.cs
    ContentIndexRecord.cs
    Migrations/
```

`BocchiDbContext` 建议继承 `IdentityDbContext<BocchiUser, IdentityRole, string>`，统一托管：

- ASP.NET Core Identity 表。
- Setup 完成状态与站点初始化信息。
- 外部登录 Provider 设置。
- Dashboard UI 偏好与主题模式默认值。
- Theme 配置与 schema cache。
- 内容索引 / 扫描摘要 / 构建运行 / 构建日志 / 构建产物 manifest 的管理投影。

边界：

- Markdown 正文、YAML frontmatter、原始媒体仍在内容空间。
- EF Core 表中可以保存内容摘要、slug、文件路径、hash、mtime、诊断信息和最近扫描结果，但不保存正文事实。
- M4 实现时需要决定是一次性迁移 M2/M3 现有 SQLite 表，还是先把新表落在 EF Core 下并逐步替换读取路径；若出现冲突，以“最终统一 EF Core 管理 Home Server 数据访问”为目标。

### 4.3 Setup / Welcome

首次启动的判定条件：

- SQLite 数据库不存在，或 EF Core migrations 未应用，或 SetupState 缺失，或没有任何 Admin 用户。

流程：

1. 进入 `/Setup`，展示 Welcome 页面与初始化检查。
2. 应用 EF Core migrations，创建 SQLite 数据库和 Identity 表。
3. 创建第一个本地账户，强制加入 `Admin` role。
4. 写入 SetupState：`CompletedAt`、`FirstAdminUserId`、`WorkspaceRoot`、`SchemaVersion`。
5. 立即关闭公开注册。
6. 自动登录第一个 Admin，进入 `/Admin`。

验收要点：

- Setup 完成前不能进入 `/Admin`。
- Setup 完成后 `/Setup` 不再显示创建账户表单。
- 第一个账户必定是 Admin。
- 后续不能通过公开注册创建普通账户。

### 4.4 用户与角色

模型：

- Bocchi 是单站点、多用户系统，不是多租户系统。
- 只有一个业务角色：`Admin`。
- 第一个账户默认加入 `Admin`。
- Admin 可以管理所有内容、设置、构建、Theme、用户与外部登录。
- 非 Admin 用户没有任何 Dashboard 权限。保留非 Admin 用户的意义，是为外部登录绑定、未来私有预览或只读场景留底层空间，但 M4 不授予功能权限。

用户管理页面：

- 用户列表：邮箱 / 用户名、是否 Admin、登录方式、状态、创建时间、最近登录时间。
- 创建用户：仅 Admin 可用；默认非 Admin，除非显式勾选 Admin。
- 禁用用户：不删除历史记录；禁用后不能登录。
- 重置密码或发起密码重设：M4 可先做本地重置流程，不引入邮件发送。
- 外部登录绑定：展示用户已绑定的 GitHub / OIDC 登录，不显示 provider secret。

### 4.5 第三方登录

设置页分为两个 provider 区块：

#### GitHub

字段：

- Enabled
- DisplayName，默认 `GitHub`
- ClientId
- ClientSecret
- CallbackPath，默认 `/signin-github`
- AllowedOrganizations，M4 可只预留字段，不强制实现组织限制

行为：

- Enabled 且 ClientId / ClientSecret 完整时，登录页显示 GitHub 按钮。
- GitHub 登录成功后，如果 email 与现有用户匹配，可由 Admin 决定是否允许自动绑定；M4 默认不自动提权。
- GitHub 外部登录不能绕过 Setup；没有 Admin 时仍必须先完成 Setup。

#### OpenID Connect

字段：

- Enabled
- DisplayName，默认 `OpenID Connect`
- Authority
- ClientId
- ClientSecret
- CallbackPath，默认 `/signin-oidc-custom`
- ResponseType，默认 `code`
- UsePkce，默认 true
- Scopes，默认 `openid profile email`
- NameClaimType
- EmailClaimType

行为：

- 配置保持 provider-neutral；Logto 只能作为示例值或帮助文案，不进入类型名、表名、路由名。
- OIDC 登录成功后不自动创建 Admin；必须由已有 Admin 授权或邀请。
- 保存配置后登录页按钮立即按 Enabled 状态变化；底层 authentication options 可通过配置缓存失效或应用重载实现。

### 4.6 Dashboard 信息架构

一级导航：

- Overview
- Content
- Build
- Preview
- Settings

二级导航：

| 一级 | 二级 | 说明 |
| --- | --- | --- |
| Overview | Status / Recent Activity / Diagnostics | 站点状态、工作区状态、最近构建、最近内容变更 |
| Content | Posts / Pages / Works / Notes / Friends / Site Settings | MVP 内容类型入口 |
| Build | Runs / Current Build / Artifacts / Download | 构建历史、实时日志、产物、zip |
| Preview | Open Preview / Route Map | 打开前台预览、查看 route 到 content 的映射 |
| Settings | Basic Info / Database / Third-party Login / Theme / Users / Workspace | 用户指定的二级分类和后续扩展位 |

桌面端：

- 左侧一级侧边栏。
- 进入某个一级分区后，内容区左侧显示二级菜单。
- 主内容区保持低到中信息密度，普通入口优先采用 App-like 单列内容列表；宽表格只用于高级管理或诊断场景。

移动端：

- 一级导航收为顶部或底部菜单。
- 二级分类显示为下拉选择框。
- 编辑器默认单栏切换：Edit / Preview / Split；窄屏不强行左右分栏。

### 4.7 UI 风格与主题

M4-T01 必须先完成 UI 方向确认，避免边做功能边反复推翻界面。

已确认的设计要求见 `Docs/Milestones/M4/UI-Design.md`。核心结论：

- 风格：柔和、低到中信息密度、App-like 的个人发布工具，而不是 GitHub 风格仓库面板、企业 CMS、硬核运维工作台或 landing page。
- 受众：18-35 岁个人 Blog / 主页创作者，兼顾 PC 工作习惯和移动端 App 使用习惯。
- Logo：左上角仅显示文字 `Bocchi`，不放 icon、头像、吉祥物或图形标。
- Dashboard 外观：右上角使用紧凑下拉，例如 `外观: Auto`，不使用占宽的 `Light / Dark / Auto` 分段控件；这里的外观切换只控制 Dashboard dark mode，不是前台业务 Theme。
- 内容列表：默认采用 App-like 单列列表，包含标题、类型 chip、路径/时间、温和状态和一个主操作；避免宽密表格。
- 发布/构建状态：普通页面使用用户能理解的检查状态和下一步动作；原始 Log 只放在高级详情。
- 颜色：Light / Dark / Auto 三态；默认 Auto。粉色必须低饱和、面积受控，与柔和蓝、淡紫、暖白和深色正文共同使用。
- 动漫灵感：只取抽象气质，不复制任何可识别角色、服装、发型、姿势、乐队标识或具体场景。
- 可访问性：键盘焦点、对比度、表单错误提示、按钮可辨识。

交付物建议：

- `Docs/Milestones/M4/UI-Design.md`：视觉方向、布局规则、组件清单、主题 token、移动端规则。
- 已确认效果图：`Docs/Milestones/M4/Assets/m4-ui-style-direction-2026-05-14.png`。
- 项目局部 skill：`.codex/skills/bocchi-ui-style/SKILL.md`，用于后续设计、生成效果图、实现和评审时恢复 UI 护栏。
- Dashboard 静态截图或 HTML prototype：Overview、Content List、Editor、Settings、Login、Setup、Preview Toolbar。
- Playwright 截图基线：桌面宽屏、普通桌面、平板、手机。

通过标准：

- UI 方向经用户确认后，才进入 M4-T06 之后的大规模页面开发。
- 如果用户明确不满意界面，不把功能页标记完成。

### 4.8 Markdown 编辑器

编辑器不是 WYSIWYG。

默认桌面布局：

- 左侧：Markdown 编辑区。
- 右侧：实时预览。
- 上方：标题、状态、保存、打开预览、查看 diff、构建提示。
- 侧边：frontmatter 表单，包括 slug、status、publishedAt、tags、category、cover 等。

移动端布局：

- Edit / Preview / Metadata 三个模式切换。
- 保存按钮固定在安全位置，避免键盘遮挡。

实现边界：

- Markdown 渲染复用现有 Markdig pipeline，避免编辑器预览和 Generator HTML 行为发散。
- 保存时写回内容空间文件，并触发扫描 / 索引刷新。
- 对 M3 约定的“修正性回写”提供确认 UI：用户看见 diff 后主动应用，不静默改 frontmatter。
- M4 不做多人协同编辑，不做实时协作，不做块编辑器。

### 4.9 前台实时预览与浮动工具栏

Dashboard 中提供“Open Preview”按钮：

- 在新标签页打开 `/`。
- 由于 Home Server 全站需要登录，打开后自然处于登录态下的预览模式。
- 未登录时访问 `/` 跳转到登录页，登录后回到原预览 URL。

Preview Toolbar：

- 固定在页面边缘，默认轻量收起，展开后显示：
  - Preview 状态标识。
  - 当前 route。
  - 返回 Dashboard。
  - 重新构建 / 刷新预览。
  - 如果 route 映射到内容：编辑文章 / 编辑页面 / 编辑作品。
- 工具栏样式由 Home Server 控制，不要求 Theme 适配。
- 工具栏资源走 `/Admin/assets/preview-toolbar.*` 或类似受保护路径。

Route Map：

- M3 ContentGraph 已有站点 URL 规则，M4 增加 route → content 的查询能力。
- 文章、页面、作品详情页必须能跳回对应编辑页。
- 首页、列表页、搜索页等聚合页无法唯一定位内容时，只显示 Dashboard 入口。

与 M5 的关系：

- M4 负责 Preview Host、鉴权、toolbar、route map 和编辑跳转。
- M5 负责默认 SvelteKit Theme 的真实页面视觉。
- 如果 M4 时没有可用 Theme，Preview Host 可以显示“当前没有可用 Theme”的受保护状态页，但不实现一套临时前台主题。

## 8. 风险与护栏

| 风险 | 护栏 |
| --- | --- |
| UI 边做边推翻导致反复返工 | M4-T01 作为强制阶段，先确认风格与核心页面，再开发主要功能；后续使用 `Docs/Milestones/M4/UI-Design.md` 与 `.codex/skills/bocchi-ui-style/SKILL.md` 复位 |
| Identity 接进 Blazor 时把登录逻辑塞进 interactive component | 使用 ASP.NET Core Identity 推荐模式，登录/回调/登出走服务端 Identity 端点；Dashboard 组件只消费认证状态 |
| EF Core 与早期手写 SQLite 并存时间过长 | M4 明确以 EF Core 为最终 Home Server 数据访问边界，新增表不再走手写 migrator |
| 第三方登录设置写死 Logto | 数据模型使用 OpenID Connect 通用字段；Logto 只作为示例说明 |
| Preview 目录被当成公开静态目录暴露 | 输出目录位于 web root 外，通过授权端点返回；HTML 注入 toolbar，非 HTML 只做受控文件响应 |
| 非 Admin 用户意外获得权限 | 全部 Dashboard 操作统一走 Admin policy，非 Admin 不给内容/设置/构建权限 |
| Markdown 编辑器预览与构建输出不一致 | 编辑器预览复用现有 Markdown pipeline；Theme 级差异只在 M5 Theme 中处理 |
| 普通用户被构建日志和诊断信息劝退 | Overview / Publish 首页只展示人能理解的检查状态；raw log、manifest、artifact tree 放入高级详情 |
| 移动端二级导航失控 | 桌面二级菜单、移动端下拉选择框作为验收项写入截图 smoke；主内容列表默认可折叠为手机单列 |

## 9. 参考调研

- [ASP.NET Core Blazor authentication and authorization](https://learn.microsoft.com/en-us/aspnet/core/blazor/security/?view=aspnetcore-10.0)：Blazor 使用 ASP.NET Core 的认证机制，服务端授权可以保护页面与组件；Identity 相关 UI 需要按 ASP.NET Core Identity 的推荐方式接入。
- [Role-based authorization in ASP.NET Core MVC](https://learn.microsoft.com/en-us/aspnet/core/mvc/security/authorization/roles?view=aspnetcore-10.0)：角色授权通过 Identity / claims 暴露，M4 只保留 `Admin` 业务角色。
- [External provider authentication in ASP.NET Core Identity](https://learn.microsoft.com/en-us/aspnet/core/security/authentication/social/?view=aspnetcore-10.0)：Identity 支持外部 OAuth 登录；多个 provider 通过 authentication builder 注册。
- [Configure OpenID Connect Web authentication in ASP.NET Core](https://learn.microsoft.com/en-us/aspnet/core/security/authentication/configure-oidc-web-authentication?view=aspnetcore-10.0)：通用 OIDC Provider 应使用标准 OIDC middleware 配置。
- [EF Core SQLite provider](https://learn.microsoft.com/en-us/ef/core/providers/sqlite/)：SQLite 是 EF Core 官方 provider，可用于本地 Home Server 应用数据库。
- [Static files in ASP.NET Core](https://learn.microsoft.com/en-us/aspnet/core/fundamentals/map-static-files?view=aspnetcore-10.0)：受保护静态文件应避免裸放在 web root 公开访问，可以在授权之后通过受控静态文件或 action/file endpoint 返回。

## 10. 决策收束

- M4 选择“新增 Home Server 应用表进入 EF Core，保留 M2/M3 已有内容扫描与构建状态表的读取路径”：Identity、Setup、Dashboard 设置、第三方登录、Theme 配置走 `BocchiDbContext` 与 EF Core migration；内容事实和 M3 构建状态仍按既有 pipeline 承担。这个边界避免把 M4 变成数据库大迁移，同时不再扩大手写 SQL 的新业务边界。
- UI 视觉基线已经固定到 `Docs/Milestones/M4/UI-Design.md` 与项目局部 `.codex/skills/bocchi-ui-style/SKILL.md`。M4 后续 UI 工作默认沿用该基线，不再回到临时后台壳。
- 第三方登录 secret 使用 ASP.NET Core Data Protection 保护后落库；登录页只显示已启用且配置完整的 GitHub / OIDC provider。Logto 仅作为 OIDC 示例，不进入类型名、路由名或数据模型。
- 非 Admin 用户保留在 Identity 中，可被创建、禁用、绑定/解绑外部登录，但默认没有 Dashboard 权限。Users 页面防止移除或禁用最后一个可用 Admin，避免把 Home Server 锁死。
